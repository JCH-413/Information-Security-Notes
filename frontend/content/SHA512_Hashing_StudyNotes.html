Consider Alice sending a message to Bob over an unprotected network. An attacker Eve can exploit four categories of weakness:

Hash Functions — Theory & Requirements

📖 Formal Definition

A hash function H accepts a variable-length block of data M as input and produces a fixed-size hash value h:

h = H(M)

The output h is also called: hash code, hash value, or message digest. It can be thought of as a fixed-size fingerprint of a variable-sized message. The hash function itself is assumed to be public (no secret key required).

✅ Formal Requirements for Hash Functions

RequirementTechnical DefinitionPlain English
Variable Input Can be applied to any sized message M Works on any file, message, or data regardless of size
Fixed Output Produces fixed-length output h Output is always the same size no matter the input size
Efficiency Easy to compute h = H(M) for any M Computing the hash is fast and computationally cheap
One-Way Property (Pre-image Resistance) Given h, it is computationally infeasible to find x such that H(x) = h You cannot reverse the hash to get the original message
Second Pre-image Resistance Given x, it is infeasible to find y (y ≠ x) such that H(y) = H(x) You cannot find a different message that gives the same hash as a known message
Collision Resistance It is infeasible to find any x, y (x ≠ y) such that H(x) = H(y) It is practically impossible to find any two different messages with the same hash
Collision resistance is the strongest requirement and the hardest to achieve. Breaking collision resistance breaks the entire hash algorithm's trustworthiness.

🌊 Avalanche Effect

The avalanche effect means that a message digest depends on ALL bits in the input message. Even the tiniest change in input causes a completely different (unpredictable) hash output.

Message: "The quick brown fox jumps over the lazy dog"
SHA-1: 2fd4e1c67a2d28fced849ee1bb76e7391b93eb12

Changed: "The quick brown fox jumps over the lazy dogs" (added 's')
SHA-1: 8de49570b9d941fb26045fa1f5595005eb5f3cf2

→ Completely different output despite only one character change

Real-world use: If a downloaded file is tampered with in even one bit, its hash will not match the published hash, immediately revealing the modification.

SHA-512 Overview

📖 What SHA-512 Does

  • Input: A message of maximum length < 2¹²⁸ bits (any size up to this limit)
  • Output: A 512-bit (64-byte) message digest
  • Processing unit: Input processed in 1024-bit blocks
  • Internal state: Eight 64-bit registers (512 bits total)
  • Rounds per block: 80 rounds

🔄 SHA-512 — Five-Step Overview

Step 1
Append Padding Bits — pad message so total length ≡ 896 (mod 1024)
↓
Step 2
Append Length — append 128-bit integer = original message length
↓
Step 3
Initialize Hash Buffer — set eight 64-bit registers to fixed constants
↓
Step 4
Process 1024-bit Blocks — 80 rounds per block using message words Wt and constants Kt
↓
Step 5
Output — after N blocks, the final 512-bit hash = concatenation of registers a–h

After Steps 1 and 2, the message length is always an exact multiple of 1024 bits: N × 1024 bits, processed as N blocks M₁, M₂, … M_N.

05

Steps 1 & 2 — Padding & Length Append

STEP 1 — Append Padding Bits

Goal: Make message length ≡ 896 (mod 1024)

  • Padding is always added — even if the message is already the right length
  • Number of padding bits: between 1 and 1024 (never 0)
  • Format: a single 1-bit followed by as many 0-bits as needed
  • Why 896? Because 896 + 128 (length field) = 1024 — making the total a multiple of 1024
Padding format: 1 0 0 0 0 0 ... 0
One mandatory '1' bit + necessary zeros to reach 896 mod 1024
STEP 2 — Append Length

Goal: Record the original message length

  • A 128-bit block is appended to the padded message
  • This block is an unsigned 128-bit integer
  • It contains the length of the original message (before padding) in bits
  • After this step, the total message length = N × 1024 bits (exact multiple)

Why append the length? This prevents length-extension attacks and ensures different messages of the same content but different lengths produce different hashes.

|←——————————— L bits (original) ———→||← padding →||← 128 bits →|

|——————1024——————|——————1024——————| ... |——————1024——————|

Step 3 — Initialize Hash Buffer

STEP 3 — Initialize Hash Buffer

Goal: Set up the eight 64-bit working registers

  • A 512-bit buffer holds intermediate and final results
  • Represented as eight 64-bit registers named a, b, c, d, e, f, g, h
  • Initialized to the first 64 bits of the fractional parts of the square roots of the first eight prime numbers (2, 3, 5, 7, 11, 13, 17, 19)
  • This is also called the IV (Initialization Vector) or H₀

🔢 Initial Register Values (SHA-512) — MUST MEMORIZE FORMAT

a = 6A09E667F3BCC908    ← first 64 bits of fractional part of √2
b = BB67AE8584CAA73B    ← fractional part of √3
c = 3C6EF372FE94F82B    ← fractional part of √5
d = A54FF53A5F1D36F1    ← fractional part of √7
e = 510E527FADE682D1    ← fractional part of √11
f = 9B05688C2B3E6C1F    ← fractional part of √13
g = 1F83D9ABFB41BD6B    ← fractional part of √17
h = 5BE0CD19137E2179    ← fractional part of √19

These are nothing-up-my-sleeve numbers — using mathematical constants as initialization values proves the algorithm has no hidden backdoors or weaknesses built into those constants.

🧠 Beginner Explanation — Why These Specific Values?

The designers needed fixed starting values that are clearly not chosen to create weaknesses. By using the first 64 bits of √2, √3, √5, etc. (irrational, unpredictable numbers), they demonstrate complete transparency: anyone can verify these values with a calculator. There's no hidden weakness, no backdoor — they are mathematically derived constants.

Step 4 — Process 1024-Bit Blocks (80 Rounds)

STEP 4 — Process Message in 1024-Bit Blocks

Goal: Compress each 1024-bit block into the 512-bit buffer through 80 rounds

  • The core compression module is labeled F
  • F consists of 80 rounds (numbered 0 to 79)
  • Each round takes the 512-bit buffer (abcdefgh) as input and updates its contents
  • At the start of each block's processing, the buffer is initialized to Hi-1 (the previous block's output)
  • Each round t uses a 64-bit message word Wt derived from the current 1024-bit block Mi
  • Each round also uses an additive constant Kt (t = 0 to 79) — derived from cube roots of the first 80 prime numbers
  • The constants Kt eliminate regularities in input data and prevent attacks that exploit patterns
  • After round 79, the output is added to Hi-1 to produce Hi (feed-forward mechanism)

The feed-forward addition (adding output of round 80 to the input Hi-1) is a critical security feature — it prevents certain algebraic attacks and ensures each block's output depends on ALL previous blocks.

Round Internals — Functions & Variable Updates

📖 What Happens Inside Each Round

Each of the 80 rounds performs the same sequence of operations on the eight working variables (a–h), computing two temporary values (temp1 and temp2) and updating all eight variables.

⚙ The Six Computations Per Round

Compute S1 (Sigma-1 — based on e)
S1 = ROTR(e, 14) ⊕ ROTR(e, 18) ⊕ ROTR(e, 41)
ROTR(x, n) = rotate x right by n bits. ⊕ = XOR
Uses register e. Three rotations, then XOR together.
Compute Ch (Choose Function — based on e, f, g)
Ch(e, f, g) = (e ∧ f) ⊕ (¬e ∧ g)
∧ = AND, ¬ = NOT, ⊕ = XOR
Where e=1: choose bit from f. Where e=0: choose bit from g.
Ch = (e AND f) XOR (NOT e AND g)
Compute temp1
temp1 = h + S1 + Ch(e,f,g) + K[t] + W[t]
All additions are mod 2⁶⁴ (64-bit modular addition)
K[t] = round constant, W[t] = message word for this round
Compute S0 (Sigma-0 — based on a)
S0 = ROTR(a, 28) ⊕ ROTR(a, 34) ⊕ ROTR(a, 39)
Same pattern as S1 but applied to register a with different rotation amounts
Compute Maj (Majority Function — based on a, b, c)
Maj(a, b, c) = (a ∧ b) ⊕ (a ∧ c) ⊕ (b ∧ c)
Output bit = 1 if at least two of the three inputs are 1 (majority vote)
Maj = (a AND b) XOR (a AND c) XOR (b AND c)
Compute temp2
temp2 = S0 + Maj(a,b,c)
64-bit modular addition

🔄 Variable Update After Each Round

After computing temp1 and temp2, all eight variables shift and update:

BEFORE ROUND

a, b, c, d, e, f, g, h

AFTER ROUND

a = temp1 + temp2
b = old a
c = old b
d = old c
e = old d + temp1
f = old e
g = old f
h = old g

Most variables simply shift down (b←a, c←b, etc.). Only a and e get new computed values. This shift structure ensures every bit of every register influences future rounds — diffusion.

🧠 Intuition Behind the Functions

FunctionWhat It DoesWhy It's Used
ROTR (Rotate Right)Shifts bits right, wrapping aroundMixes bits in a non-linear way — diffusion
Ch (Choose)e selects between bits of f and gNon-linear selection — confusion
Maj (Majority)Output is the majority bit of a, b, cNon-linear combination — confusion
XOR (⊕)Bitwise exclusive ORMixes bits with no key cancelation
AND (∧)Bitwise ANDCreates non-linear interaction between bits

SHA-512 Worked Example — Input "abc"

📖 The Problem

Compute SHA-512 of the string "abc" (3 ASCII characters = 24 bits). We will trace through every preparatory step and the structure of Round 0.

Convert "abc" to Binary
'a' = 01100001 (ASCII 97)
'b' = 01100010 (ASCII 98)
'c' = 01100011 (ASCII 99)

Message bits: 01100001 01100010 01100011 (24 bits total)
Step 1 — Append Padding

Current length: 24 bits. Need length ≡ 896 (mod 1024).

24 + 1 (the mandatory '1' bit) = 25 bits
25 + k = 896 → k = 871 zeros

Result: 01100001 01100010 01100011 1 [871 zeros...]
Total after padding: 896 bits
Step 2 — Append Length

Original message length = 24 bits = 0x18 in hexadecimal.

Append: 000...00011000 (128 bits, representing decimal 24)

Final padded message: 896 + 128 = 1024 bits = 1 complete block ✓

For "abc", we have exactly ONE 1024-bit block after padding.

Step 3 — Initialize Hash Values (Working Variables)
a = 6a09e667f3bcc908
b = bb67ae8584caa73b
c = 3c6ef372fe94f82b
d = a54ff53a5f1d36f1
e = 510e527fade682d1
f = 9b05688c2b3e6c1f
g = 1f83d9abfb41bd6b
h = 5be0cd19137e2179
Round 0 — Step-by-Step Calculations

This pattern repeats for all 80 rounds. Only W[t] and K[t] change each round.

1. Compute S1:
S1 = ROTR(e, 14) ⊕ ROTR(e, 18) ⊕ ROTR(e, 41)
e = 510e527fade682d1

2. Compute Ch(e, f, g):
Ch = (e AND f) XOR (NOT e AND g)
e = 510e527fade682d1
f = 9b05688c2b3e6c1f
g = 1f83d9abfb41bd6b

3. Compute temp1:
temp1 = h + S1 + Ch + K[0] + W[0]
h = 5be0cd19137e2179
K[0] = 428a2f98d728ae22 (cube root of 2nd prime)
W[0] = first 64 bits of the padded message block

4. Compute S0:
S0 = ROTR(a, 28) ⊕ ROTR(a, 34) ⊕ ROTR(a, 39)
a = 6a09e667f3bcc908

5. Compute Maj(a, b, c):
Maj = (a AND b) XOR (a AND c) XOR (b AND c)
a = 6a09e667f3bcc908
b = bb67ae8584caa73b
c = 3c6ef372fe94f82b

6. Compute temp2:
temp2 = S0 + Maj

7. Update variables:
a = temp1 + temp2
b = 6a09e667f3bcc908 (old a)
c = bb67ae8584caa73b (old b)
d = 3c6ef372fe94f82b (old c)
e = a54ff53a5f1d36f1 + temp1 (old d + temp1)
f = 510e527fade682d1 (old e)
g = 9b05688c2b3e6c1f (old f)
h = 1f83d9abfb41bd6b (old g)

This pattern repeats for rounds 1–79. After round 79, the final values of a–h are added (mod 2⁶⁴) to the initial values of H₀ to produce H₁ = the final hash.

MD5 — Comparison & Context

📖 MD5 Background

  • Designed by Ronald Rivest — the "R" in RSA
  • Latest in the MD series: MD2 → MD4 → MD5
  • Produces a 128-bit hash value
  • Until recently, was the most widely used hash algorithm
  • Specified as Internet standard RFC 1321
  • Now considered vulnerable — should NOT be used for security-sensitive applications

⚙ MD5 Process Overview

  1. Pad message so its length is 448 mod 512
  2. Append a 64-bit length value to the message
  3. Initialise a 4-word (128-bit) MD buffer (A, B, C, D)
  4. Process message in 16-word (512-bit) blocks: using 4 rounds of 16 bit operations on message block and buffer
  5. Add output to buffer input to form new buffer value
  6. Output: the final 128-bit buffer value

⚠ MD5 Known Attacks & Weaknesses

  • Berson (1992): Attacked any single round using differential cryptanalysis — but couldn't extend the attack beyond one round
  • Boer & Bosselaers (1993): Found a "pseudo-collision" — similar to a collision but using different initial values — also not extendable to full MD5
  • Dobbertin (1996): Created actual collisions in the MD5 compression function — however the standard initial constants prevented this from being exploited directly
  • Conclusion: MD5 looks vulnerable — real-world collision attacks on full MD5 have since been demonstrated; do not use for certificates or security-sensitive hashing
