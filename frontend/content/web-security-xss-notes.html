01

Introduction to XSS

📖 What This Topic Covers

Cross-Site Scripting (XSS): how the attack works, two practical attacks (Add Friend and Modify Profile), writing a self-propagating XSS worm, countermeasures, Content Security Policy (CSP), and attack generalization.

🌐 The Samy Worm (2005)

In 2005, Samy Kamkar generated a worm on the social media website myspace.com that carried a payload. As victims clicked on the payload, two things happened:

  • Samy was added as a friend on the victim's profile.
  • The victims' profiles were updated with the text "Samy is my hero".

This worm victimized one million users within 20 hours on October 4, 2005.

🗳️ The votehillary.org Case (2008)

In 2008 there was a US election with Obama and Hillary as candidates. Hillary had an election-campaign website at votehillary.org, which became a victim of an XSS attack.

The attack changed the content of votehillary.org — the URL stayed the same, but the content shown was Obama's website.

02

How XSS Works & XSS vs. CSRF

⚙️ What is Cross-Site Scripting (XSS)?

XSS is a form of malicious code injection into the victim's system through a website or web server.

  • The victim sends a single-site request to a legal and authenticated server.
  • But the attacker injects code into the server's response due to a server vulnerability.
  • As the browser sees code in the HTTP response, the browser immediately executes the code.

⚖️ Difference between CSRF and XSS

Both attacks abuse the victim's authenticated session, but the trust direction differs: in CSRF the attacker forges a request from a malicious site to a trusted server; in XSS malicious code is injected into and executed within a trusted page itself.

📑 Types of XSS Attack

  • Non-persistent XSS Attack
  • Persistent XSS Attack (e.g., the Samy worm example)
03

Damages Caused by XSS

🖥️ Attack the Server

The attacker sends an HTTP request to the server on behalf of the user. The server responds and updates the user's profile according to the attacker's HTTP request.

🎨 Web Defacing

JavaScript code runs inside the page and has the privilege to access elements of the page. The code can change, modify, and delete the elements.

The Obama and Hillary XSS case was web defacing on the client side.

04

Attack 1: Add Friend (Persistent XSS)

🎯 Objective

The objective of this attack is to add the attacker into the victim's friend list as the victim visits the attacker's profile.

🔍 Investigation

The attacker investigates the GET method request before launching the attack (the "Add Friend" HTTP request).

🔑 How the Attacker Gets Tokens and Timestamp

  • The developer uses a global variable for the token and timestamp inside the web page.
  • When the attacker's malicious code is inside the web page, the code can easily access the global variables of token and timestamp.
  • Example: Elgg.security.token.__elgg_token

This is a key difference from CSRF: because XSS code runs inside the trusted page, the anti-CSRF token defense is defeated.

🛠️ Construct the URL and Ajax Code

The attacker constructs the attack URL and writes Ajax code to send the forged request, then completes the code and places it in plaintext mode in the attacker's profile.

💥 Reaction of the Attack

As the victim visits Samy's profile, the code triggers and Samy is added to the friend list of the victim.

05

Attack 2: Modify Profile

🎯 Objective

The objective of this attack is to edit the victim's profile as the victim visits the attacker's profile.

🔍 Investigate the Request

The attacker investigates the Edit Profile HTTP request before launching the attack.

🛠️ Construct the Data and Ajax Code

The attacker constructs the data and the Ajax code, completes the code, and places it in plaintext mode in the attacker's profile. When the victim visits, the profile-editing request triggers.

06

Writing a Self-Propagating XSS Worm

📈 Spreading the Worm Exponentially

A self-propagating worm spreads exponentially: each infected profile infects every visitor, who then spreads it further.

🧬 Two Ways to Make an Identical Copy of Malicious Code

  • By yourself (Self-Reproducing)
  • Using external help

♻️ By Yourself (Self-Reproducing / Quine)

The "by yourself" approach means putting a copy of your code inside your code.

Mechanism: the attacker writes code that prints its own source code without reading any external file. This technique is also called a "Quine".

🌍 Using External Help

The "using external help" approach means saving your code somewhere else and loading it when needed (e.g., the worm code file xssworm.js).

Another way: load the code inside an element of the page as content. As the web page loads in the browser, the browser has an API to access the element by id and retrieve the code.

07

Countermeasures

⚠️ Fundamental Problem of XSS

Mixing data and code is hard to separate because both have the same format (like a string). This is the fundamental problem, because the server cannot sense whether data coming from a user has malicious code inside it or not.

🧹 1st Solution: Apply Filter on Data, Extract Code

Different filters can be applied to data coming from users to separate data and code. But hackers are creative and can counter the filters — as Samy demonstrated:

  • Samy's strategy: <div style="background:url('javascript:alert(1)')">
  • Filter 1: MySpace filter blocks tags like <script>, <head>, <body> etc.
  • Samy's strategy: <div style="background:url('javascript(1)')">
  • Filter 2: MySpace strips out the word "javascript" from anywhere.
  • Samy's strategy: eval('xmlhttp.onread' + 'ystatechange = callback');
  • Filter 3: MySpace strips out the word "onreadystatechange".

The filters are not effective, because hackers have a creative nature and can counter them.

🔄 2nd Solution: Turning Code into Data

The 2nd solution is to turn code into data with the help of special characters. So a full dictionary of special characters is required to convert code into data.

But the problem is that this solution can also be bypassed easily.

📦 3rd Solution: Treating Data Only as Data (Sandbox)

The 3rd solution is to use an iframe as sandboxing functionality. This solution tells the browser to treat data = [data + code] as data, even if it contains code.

Setting the sandbox — options of privilege level:

  • allow-scripts
  • allow-same-origin
  • allow-forms
  • allow-modals
  • allow-top-navigation
08

CSP: Content Security Policy

📖 What is CSP?

CSP is another countermeasure for XSS.

🕵️ The Confused Deputy / Origin Problem

CSP helps the browser know about the deputy/origin of code and data. CSP forces the code to be labelled with respect to its deputy/origin/source.

🏷️ Forcing Code to be Labelled

There are two ways to include JavaScript code inside HTML:

  • Inline approach
  • Linked approach

The source of the linked approach must be labelled, because code may be coming from some other resources. Once a source is labelled, CSP grants privilege to that code.

⚙️ Setting Content Security Policy

  • Setting CSP in PHP code — used for multiple headers.
  • Setting CSP on Apache — used for a single type of header.
09

Attack Generalization

📱 Attack on HTML5-based Mobile Applications

Mobile phones have different operating systems and architectures, like Android and iOS. The application developer has to write separate code for both Android and iOS applications.

  • So developers convert Android and iOS applications into web-based applications that support both platforms.
  • Vulnerabilities like CSRF and XSS come into mobile applications due to these web-browser-based applications.

💉 Code Injection into Mobile

Because HTML5-based mobile apps render web content, malicious code can be injected into the mobile application — including attacks via QR code, where scanning a malicious QR code delivers the payload.

📝 Common Exam Questions
  • What is Cross-Site Scripting (XSS)? → Malicious code injection into a trusted server's response that the browser then executes
  • Describe the Samy worm. → 2005 MySpace persistent XSS worm; added Samy as friend and posted "Samy is my hero"; infected 1M users in 20 hours
  • Differentiate XSS from CSRF. → XSS uses a single-site request with code running inside the trusted page; CSRF uses a cross-site forged request
  • What are the two types of XSS? → Non-persistent and persistent
  • How does XSS defeat the anti-CSRF token? → Malicious code runs inside the page and reads the token/timestamp global variable (e.g., Elgg.security.token.__elgg_token)
  • What is a Quine in the context of XSS worms? → Code that reproduces (prints) its own source without reading an external file
  • What is the fundamental problem of XSS? → Data and code share the same string format, so the server cannot separate malicious code from data
  • List the three filtering/defense solutions and their weakness. → Filter data/extract code (bypassable), turn code into data (bypassable), iframe sandbox (treat data+code as data)
  • What does CSP do? → Forces code to be labelled by origin/source so the browser can apply privileges; solves the confused-deputy problem
  • Why are HTML5-based mobile apps vulnerable to XSS/CSRF? → They are web-browser-based apps built for cross-platform support, inheriting web vulnerabilities, including injection via QR code
