03

WHERE, UPDATE & Comments

🔎 Conditions: WHERE

SELECT * FROM mytable WHERE Name='Alice'; returns only Alice's row
SELECT * FROM mytable WHERE Name='Alice' OR Salary>8000; returns Alice (id 1) and David (id 4)

⭐ Special Condition (Always True)

SELECT * FROM mytable WHERE 1=1; 1=1 is always true → returns ALL rows

This "always-true" trick is the heart of many SQL injection attacks — see Login Without Password.

✏️ UPDATE Statement

UPDATE mytable SET Salary=9999 WHERE Name='Alice'; Alice's salary becomes 9999

💬 Comments in SQL

Three comment styles — important for injection, since attackers use comments to discard the rest of a query:

  • SELECT * FROM employee; # comment (hash)
  • SELECT * FROM employee; -- comment (double dash)
  • SELECT * FROM /* In-line comment */ employee; (inline)
04

How a Web App Interacts with the Database

🔗 The Login Flow

The browser submits a username and password through a login form. The server builds an SQL query against the credential table to authenticate the user.

SELECT name, eid, salary, email
FROM credential
WHERE name='alice' and Password='seedalice';

⚠️ The Vulnerable Server Code

The application server takes the user input and concatenates it directly into the SQL string — this is the root of the vulnerability:

$input_uname = $_GET['username'];
$input_pwd  = $_GET['Password'];

$sql = "SELECT name, eid, salary, email
    FROM credential
    WHERE name='$input_uname' and Password='$input_pwd'";
$result = $conn->query($sql);

Because the user input is placed inside the query as code, a crafted input can change the query's meaning.

05

SQL Injection: Login Without Password

🔓 First Case — Known Username (Alice)

The attacker types this into the username field:

Username: alice' # single quote closes the name string; # comments out the rest

The resulting query becomes:

SELECT ... WHERE name='alice' #' and Password='...' the password check is commented out

Result: the attacker logs in as Alice and sees her full profile (Employee ID 10000, Salary 20000, SSN 10211002, etc.) without knowing her password.

🔓 Second Case — Unknown Username

Objective: you don't know any name — can you still log in? The attacker types into the username field:

Username: a' OR 1=1 # 1=1 is always true; # comments out the password check

The OR 1=1 makes the WHERE clause always true, so the query returns a user (typically the first row) and logs the attacker in without any valid credentials.

Both attacks rely on the same idea: inject a single quote to break out of the data string, then alter the query logic (OR 1=1) or remove the password check (#).
06

Modify Database Using SQL Injection

💰 Change Your Own Salary

The profile-edit page runs an UPDATE statement. Alice (Employee ID 10000, salary 20000) injects into the NickName field:

NickName: alice', Salary='1000000 closes the nickname value and adds a Salary assignment

The UPDATE now also sets her salary, raising it far above 20000.

💸 Change Someone Else's Salary

Target: Ted (Employee ID 50000, salary 110000). Inject into the NickName field a clause that re-targets the WHERE:

NickName: ', Salary=1 WHERE name='Ted' # sets Ted's salary to 1; # comments out the original WHERE

The attacker overwrites the WHERE clause to point at Ted, changing his salary to 1.

07

Running Multiple / Arbitrary SQL Statements

📑 Run Multiple SQL Statements

MySQL can run several statements separated by semicolons in one line:

mysql> select 1; select 2; select 3; each statement returns its own result set

⚙️ Run an Arbitrary SQL Statement

Objective: use the SQL injection vulnerability to run any statement. The attacker appends a second statement after a semicolon in the username field:

Username: alice'; select 1 # ends the SELECT, then runs an arbitrary statement

In principle this lets the attacker run a destructive statement such as a DROP or DELETE.

🛡️ Countermeasure of the Database (Why It Often Fails)

The database will block multiple SQL statements. This countermeasure is actually enforced in the PHP code: PHP uses an API to invoke the database, and that API supports only one single SQL statement.

So the "run arbitrary multiple statements" attack is stopped by the single-statement API — but single-statement injections (login bypass, salary change) still work.

08

Similarity With Other Code Injection

🔁 The Common Pattern

All code-injection attacks share the same root cause: untrusted user data is mixed with trusted code, and a parser then treats part of the data as code to execute.

🖥️ System Injection

Untrusted user data is mixed with a trusted command name. The shell parser splits it into Data' and Command', and the injected command' gets executed.

📜 JavaScript Injection

Also known as Cross-Site Scripting (XSS) — injected script is treated as code and executed by the browser.

🗄️ SQL Injection

Injected text inside a data field is parsed as part of the SQL statement and executed by the database engine.

09

Countermeasures

🔄 Turning Code into Data (Encoding Special Characters)

Encode special characters so they are treated as plain data rather than code:

  • Apache's configuration: magic_quotes_gpc = On in php.ini.
  • PHP's solution: mysqli::real_escape_string()
$eid = $mysqli->real_escape_string($_GET['EID']);
$pwd = $mysqli->real_escape_string($_GET['Password']);
$sql = "SELECT Name, Salary, SSN FROM employee
    WHERE eid='$eid' and password='$pwd'";

✅ Separate Code and Data — Prepared Statements (Best Solution)

Turning code into data or applying a filter is not reliable, because an attacker can easily bypass both. The better solution is to separate code and data and pass each through a separate channel to prevent code injection. Most SQL engines provide this (originally for performance) as prepared statements.

PREPARE stmt FROM "SELECT * FROM mytable WHERE name = ? or age > ?";
SET @a='Alice';
SET @b=20;
EXECUTE stmt USING @a, @b; ? placeholders are bound as pure data

🧩 Prepared Statement Program (PHP)

Vulnerable approach vs. prepared-statement approach:

Vulnerable:
$sql = "SELECT name, local, gender FROM USER_TABLE
    WHERE id=$id AND password='$pwd'";
$result = $conn->query($sql);
Using prepared statement:
$stmt = $conn->prepare("SELECT name, local, gender FROM USER_TABLE
    WHERE id = ? and password = ? ");
$stmt->bind_param("is", $id, $pwd);
$stmt->execute();
$stmt->bind_result($bind_name, $bind_local, $bind_gender);
$stmt->fetch();

The query structure is sent first; the data is bound afterward, so injected text can never change the query's meaning.

📝 Common Exam Questions
  • What is the fundamental cause of SQL injection? → User-supplied data is concatenated into the SQL query as code, so the parser executes injected SQL
  • Write the injection input to log in as a known user without a password. → alice' # (the # comments out the password check)
  • Write the injection input to log in when you don't know any username. → a' OR 1=1 # (always-true condition)
  • How can an attacker change another user's salary via injection? → Inject ', Salary=1 WHERE name='Ted' # into an updatable field to overwrite the UPDATE's WHERE clause
  • What is the role of the # in these attacks? → It comments out the remainder of the original query (e.g., the password condition)
  • How does the database/PHP block running multiple statements? → The PHP query API supports only a single SQL statement per call
  • Name three code-injection attacks and their common cause. → System injection, JavaScript injection (XSS), SQL injection — all mix untrusted data with trusted code
  • Why are filtering and encoding special characters not reliable? → Attackers can craft inputs that bypass filters/encoding
  • What is the best countermeasure and why? → Prepared statements, because code and data travel in separate channels so injected text can't alter the query structure
  • Outline the PHP prepared-statement steps. → prepare() with ? placeholders, bind_param(), execute(), bind_result(), fetch()
