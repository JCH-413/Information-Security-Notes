The web architecture is made up of three components that work together to deliver content to users.

  • Browser — the client that requests and renders content.
  • Web Application Server — processes requests and generates responses.

Web Browser: HTML, CSS & Rendering

📄 HTML

HTML provides the basic data part — it defines the structure of the web document and the content within it.

HTML and CSS are often called markup languages rather than programming languages, because at their core they provide markup for documents with very little dynamism.

🎨 CSS — Cascading Style Sheet

CSS handles the presentation part. It declares various styles for the contents on the web document.

It tells the browser how to render and deal with this content.

HTML defines structure and content; CSS declares styles. Neither provides dynamic behavior on its own — that role belongs to JavaScript.
03

Plug-ins vs. Extensions

🔌 Plug-ins

Browsers have limited functionality, but people want to do additional work — so plug-ins help handle specific types of content.

A browser plug-in is a binary piece of code that manages Internet content a browser is not designed to process.

  • Example: to open a PDF on a website you may need an Adobe Acrobat Reader plugin. Without it, the browser does not understand how to open the PDF.
  • Plug-ins lie inside the protected gate of the browser — they are like part of the browser, so they can access memory directly.

⚠️ Vulnerability of Plug-ins

Plug-ins can be a source of vulnerability because they are developed by 3rd parties, so they might come from the attacker's side. An attacker can embed malicious code inside a plug-in, and plug-ins can access memory directly.

Solution: Browser developers are adding maximum functionality into the browser, so the need for plug-ins is gradually decreasing. For example, there is no longer a need to install the Adobe Acrobat Reader plug-in, because HTML5 in the browser supports opening PDF files.

🧩 Extensions

A browser extension is a small software application that adds capability or functionality to a web browser using its own set of APIs. They can also remove unwanted website elements such as pop-up ads and features such as auto-play for online videos.

  • Extensions are written in a high-level language (scripts).
  • They talk to data and resources through APIs in the browser.
  • They do not have direct access to memory because they are outside the gate.
  • A lot of access control is involved this way.

🌐 Examples of Browser Extensions

  • AdBlock — enables content filtering and ad blocking.
  • StayFocusd — limits the amount of time the user is allowed to spend on designated websites.
  • Highly — allows the user to highlight webpage text and share it through social media or email.

Extensions lie outside the protected gate of the browser, so they cannot access memory.

Dynamic Content & JavaScript

📖 Dynamic Content

Dynamic content is a web-page or email component that changes based on user signals — i.e., the behavior, preferences, and interests of the user.

Sources of dynamic content:

  • In-session behavior — adapt content based on which pages they visit, which products they add to cart, and how long they spend on the site.
  • User data — change content based on past purchases, customer lifecycle, or past engagements.
  • User characteristics — geo-location or buyer personas.

🛠️ Technologies to Create Dynamic Content

  • Adobe Flash (animation)
  • Microsoft Silverlight
  • ActiveX
  • Java applets
  • JavaScript

These days, the most popular technology for creating dynamic content is JavaScript.

⚙️ JavaScript

JavaScript is a scripting language that enables you to create dynamically updating content, control multimedia, animate images, and implement complex features on web pages. It can calculate, manipulate, and validate data.

Unlike HTML and CSS (markup languages), JavaScript is a dynamic programming language that supports math calculations, lets you dynamically add HTML content to the DOM, fetches content from another website, and lots more.

The Risk of Running Code

Suppose you download a file and execute it — this leads to risks. But on the web we do exactly that on a daily basis: we invite code to come and execute on our computers.

🔍 How JavaScript Works

  • Client-Side Execution — JavaScript typically runs directly in the user's web browser. Every major browser has a built-in "engine" (like Chrome's V8) designed to read and execute JS code instantly.
  • Event-Driven — it "listens" for specific events (a click, a hover, a scroll) and runs a block of code (a function) only when that event occurs.
  • DOM Manipulation — JavaScript interacts with the Document Object Model (DOM), the browser's internal map of your HTML. This lets JS add, remove, or change HTML elements and CSS styles on the fly.
05

JavaScript Engine & Security Layer

⚙️ JavaScript Engine

JavaScript engines are used to compile JavaScript byte code. The engine lives inside your browser — you can also consider it a JavaScript VM.

The engines also enforce security policies, so the sandbox runs inside the JavaScript engine.

🌐 Engines by Browser

  • V8 from Google — for Chrome and Microsoft Edge
  • SpiderMonkey from Mozilla — for Firefox
  • JavaScriptCore from Apple — for Safari

All engines run bytecode and enforce security.

❓ Why Not Just Use Binary Code?

Question: why can't we just use binary code, e.g. <binary src="https://www.example.com/code.bin"></binary> (a fictional HTML tag)?

  • It is very difficult to enforce security on binary because binary runs directly on top of the CPU.
  • It deals directly with the CPU → the OS can only handle things at that stage.
  • But the OS doesn't know which binary is related to the browser and which is untrusted code — it lacks the granularity to separate them.

🛡️ The Engine Layer Solution

The Engine layer is part of the browser. Any bytecode that comes in, the browser knows whether it is from a trusted party or an untrusted party, etc. This way, we can apply access control on the behavior of bytecode.

Accessing Page, Browser & File Data

🌳 Accessing Page Content — HTML DOM

The HTML DOM (Document Object Model) treats every element as a node. You access nodes with the list of APIs provided by the browser — every element has an id.

document.getElementById("demo").innerHTML = "Hello World"; find an element by its id, then set its content

🔒 Scope of DOM Access

Once you find the reference of an element through an API, you can change the content of the same page. But you cannot modify another page — you do not have a reference to another page.

These things get complicated when a developer uses iframes inside a web page, because each iframe behaves as a web page inside a web page.

HTML Objects (examples):

  • document.cookies
  • document.images
  • document.forms
  • document.URL

🪟 Accessing Browser Data — BOM

The Browser Object Model (BOM) is centered on the window object. The browsing data belongs to the user. The server and 3rd parties can access user browsing data through APIs with user permission.

Objects (examples):

  • window.history — with back() and forward() APIs
  • window.navigator — e.g. navigator.geolocation.getCurrentPosition()

📁 How JavaScript Accesses Files

There is no API that can access a file directly from memory.

  • The file is opened in the browser when the user browses to upload it.
  • Then the reference of the file from the browser is given to the JavaScript code.

This is the beauty of access control.

Web Server & Server-Side Scripts

🖥️ Web Server

The HTTP server only responds to static content. So, Web Application Servers are used to respond with dynamic content.

The HTTP server communicates with the web application server using CGI (Common Gateway Interface), Fast CGI, or Modules (Apache).

📝 Server-Side Script

Mostly the server provides dynamic content and also creates dynamic pages. Technologies used to create dynamic content include PHP, Python, and JavaScript, etc.

For example, the Apache server knows the PHP tags and will replace the response against .date("h:i:sa") with the returned data.

📤 Returning Data on an HTML Page

  • Inline — return data directly inline on the HTML page.
  • Use a Template — used for a large number of files.

Cookies & Sessions

⚠️ Stateless Nature of the Web

Suppose a user sends an HTTP request to the server; the server processes the request and responds. But the server does not keep a record of the user.

The second time the same user sends an HTTP request, the server cannot recognize the user — whether they visited the site before or not. So this is the problem.

🍪 Cookies

Cookies are designed to keep track of the user at the user end.

  • A user sends an HTTP request; the server processes it and responds with cookies. The cookies are saved on the browser.
  • The second time the same user sends a request, the cookies are attached with the request automatically. The server then recognizes the user and treats them in a better way.

Operations: setting cookies in the HTTP response, and attaching cookies in the HTTP request (with PHP code examples).

🕵️ Web Tracking Using 3rd-Party Cookies

Third-party cookies can be used to track users across websites.

🔑 Session and Session Cookie

Cookies are stored in the user's browser, but not all user data can be stored on the user side. Suppose a shopping website stores all user data at the user side as cookies (quantity, price, size, etc.). The user's bill of $1000 is also saved as a cookie at the user end — the user could change the price. So this is a vulnerability.

The solution is that the price must be saved on the server. But then how does the server recognize which price belongs to which user? A Session ID is used for this problem.

  • The Session ID is a cookie assigned to the user at the time of login.
  • The Session ID expires when the user logs out.
  • The Session ID must be secure.
If an attacker steals the Session ID using a sniffing attack, they do not need to steal the username or password to log into the user's account.

Ajax & the Same-Origin Policy

⚡ Ajax versus Normal HTTP Requests

In a normal HTTP request, the request is sent to the HTTP server from the web page. The server processes it and responds. When the response comes back, the previous page is discarded (the one that sent the request) and a new page is loaded on the browser (this new page is the server's response).

In an Ajax request, the request is sent to the HTTP server from the web page. When the response comes back, the previous page is updated according to the server response (no full page reload).

✍️ How to Write Ajax Code

  • You need an HTTP object to send the request.
  • Ajax is asynchronous — when the user sends a request to the server, the current page does not block. The user can even use other features of the page.
  • As the server response comes back, the browser notifies you if you have registered a callback function. The user gets the server response in the callback function.

🌐 Origin Policy on Ajax

Consider having an active session of facebook.com while also using google.com, and sending a request from google.com to facebook.com:

  • HTTP Case — facebook.com responds, and the response is shown on facebook.com instead of google.com, because HTTP discards the requesting page and shows the response in a new window. This case has no vulnerability.
  • Ajax Case — facebook.com responds, and the response is shown on google.com because Ajax updates the same page. This case has a vulnerability, because the data of one website is showing on another website.

🛡️ Ajax Security Policy

In the Ajax security policy, the browser blocks cross-origin responses.

  • If a request is sent from google.com to facebook.com, facebook.com responds — but the browser blocks the response.
  • If facebook.com adds permission to display data on another site in the response header, then the browser releases the response.

⚠️ Bypassing Same-Origin Policy via DNS Rebinding

DNS rebinding is a technique used to bypass the Same-Origin Policy.

WebSocket

🔌 WebSocket

WebSocket is also used to communicate between the browser and the server. The problem with HTTP and Ajax is that the user has to establish a connection again and again to get updated data from the server.

But with WebSocket, one connection is established and the server automatically sends updated data to the client.
