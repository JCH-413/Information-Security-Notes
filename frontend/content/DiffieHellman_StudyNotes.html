01

Background & History

📖 What Is Diffie-Hellman?

The Diffie-Hellman (DH) Key Exchange is the first published public-key algorithm. It allows two parties who have never met before and share no prior secret to establish a shared secret key over a completely public, insecure channel — even while being eavesdropped upon.

Once the shared key is established, it can be used to encrypt all subsequent communication using a symmetric cipher (like AES).

📅 Key Historical Facts

  • Published by Whitfield Diffie and Martin Hellman in 1976
  • Considered the foundation of public-key cryptography
  • First publicly known work to propose the concept of a private key and corresponding public key
  • In 2006, Hellman suggested it be renamed Diffie-Hellman-Merkle to credit Ralph Merkle's contribution to the concept
  • In 1997, it was revealed that GCHQ (British intelligence) had independently discovered the same concept in 1969 — but kept it classified
  • The expired US patent 4200770 (1977) credits Hellman, Diffie, and Merkle as inventors
  • Widely used in commercial products for key exchange

🎯 Purpose & Limitation

  • Purpose: Enable two users to securely exchange a shared secret key that can then be used for symmetric encryption of messages
  • Limitation: The algorithm itself is limited to key exchange only — it does NOT encrypt messages directly
  • DH is a key-agreement protocol, not an encryption algorithm
  • It is non-authenticated by itself — neither party proves who they are (vulnerability: MITM attack)
  • The security depends on the difficulty of computing discrete logarithms

The Problem Diffie-Hellman Solves

📖 The Key Distribution Problem

Before Diffie-Hellman, all encryption was symmetric — both parties needed the same key to encrypt and decrypt. This created a fundamental chicken-and-egg problem:

  • To communicate securely, you need to share a key
  • To share a key securely, you need to already be communicating securely

The traditional solution was physical key exchange — courier delivering paper key lists. This was impractical at internet scale. DH solved this by making it possible to establish a shared secret over a public channel without ever transmitting the key itself.

❌ Before DH — The Old Way

  1. Alice and Bob need to communicate
  2. They need a shared symmetric key to encrypt
  3. They must physically meet or use a trusted courier to exchange the key
  4. If the courier is intercepted — everything is compromised
  5. Completely impractical for global internet-scale communication

✅ After DH — The New Way

  1. Alice and Bob agree on public parameters over an insecure channel (anyone can see this)
  2. Each generates their own private secret number
  3. They exchange public values derived from their secrets (not the secrets themselves)
  4. Each independently computes the same shared secret using the other's public value
  5. Eve saw all exchanges but cannot compute the shared secret
📝 Common Exam Questions
  • Long: Explain the key distribution problem in symmetric cryptography. How does Diffie-Hellman solve it?
  • Short: Why was key exchange a problem before Diffie-Hellman?
  • MCQ: What type of encryption was used before DH? → Symmetric encryption (same key for both parties)

The Paint Colour Analogy (from the Video)

The YouTube video uses a brilliant colour-mixing analogy to explain DH intuitively. This is the best way to remember the concept before tackling the math.

🎨 Paint Mixing Analogy — Step by Step
  1. Public Colour Agreement: Alice and Bob both publicly agree on a common starting colour — say Yellow. Everyone (including Eve) knows this colour.
  2. Private Secrets: Alice picks her own secret colour — Red. Bob picks his own secret colour — Cyan. Neither tells anyone their secret colour.
  3. Mix with Public Colour: Alice mixes Yellow + Red → gets Orange-tan. Bob mixes Yellow + Cyan → gets Light-blue. These mixed colours are their "public keys."
  4. Exchange Mixed Colours: Alice sends Orange-tan to Bob. Bob sends Light-blue to Alice. Eve can see both mixed colours — but cannot easily separate them back into components.
  5. Final Mixing: Alice mixes Light-blue + her secret Red → gets Yellow-brown. Bob mixes Orange-tan + his secret Cyan → gets Yellow-brown. Both arrive at the identical final colour.
  6. Eve is Stuck: Eve saw Yellow (public), Orange-tan (Alice's mix), Light-blue (Bob's mix) — but cannot determine the final Yellow-brown without knowing the private colours. Separating mixed paint is computationally equivalent to solving the discrete log problem.

The DH Algorithm — Step by Step

📖 Public Parameters Required

Two values must be agreed upon publicly (everyone including attackers can see these):

  • q (or p): A large prime number
  • α (or g): A primitive root of q — an integer such that α, α², α³, ..., α^(q-1) are all distinct values mod q

A primitive root ensures the group has maximum size — the shared secret can take any value from 1 to q-1, providing maximum security.

Step 1 — Public Agreement

Both parties agree on public parameters

Choose a large prime q and a primitive root α of q. These are published/shared openly. Anyone can see them — that is intentional.

q = large prime number   (public)
α = primitive root of q   (public)
Step 2 — Private Keys (Each Party Independently)

Each party generates their own private key

  • User A selects a random integer X_A such that X_A < q. This is A's private key.
  • User B independently selects a random integer X_B such that X_B < q. This is B's private key.
  • Both keep their X values completely secret — never shared with anyone.
X_A = Alice's private key   (secret, never transmitted)
X_B = Bob's private key   (secret, never transmitted)
Step 3 — Public Keys (Compute and Exchange)

Each party computes their public key from their private key

  • A computes: Y_A = α^(X_A) mod q — this is A's public key
  • B computes: Y_B = α^(X_B) mod q — this is B's public key
  • They exchange Y_A and Y_B over the public channel. Eve can see these values.
Y_A = α^(X_A) mod q   (Alice's public key — sent to Bob)
Y_B = α^(X_B) mod q   (Bob's public key — sent to Alice)
Step 4 — Compute Shared Secret (Each Party Independently)

Both parties independently arrive at the same shared secret

  • A computes: K = (Y_B)^(X_A) mod q
  • B computes: K = (Y_A)^(X_B) mod q
  • Both computations produce the same result K — the shared secret key
Alice: K = (Y_B)^X_A mod q
Bob: K = (Y_A)^X_B mod q
─────────────────────────────────
Both equal: K = α^(X_A × X_B) mod q   ← shared secret

🧮 Why Do They Get the Same Value?

The mathematical proof is elegant:

Alice computes: (Y_B)^X_A mod q
    = (α^X_B mod q)^X_A mod q
    = α^(X_B × X_A) mod q
    = α^(X_A × X_B) mod q

Bob computes: (Y_A)^X_B mod q
    = (α^X_A mod q)^X_B mod q
    = α^(X_A × X_B) mod q
    = α^(X_A × X_B) mod q

Multiplication is commutative: X_A × X_B = X_B × X_A
∴ Both parties compute the identical shared secret ✓

Mathematical Deep Dive — Primitive Roots & Modular Arithmetic

📖 What Is a Primitive Root?

For a prime number q, if α is a primitive root of q, then the powers of α produce every integer from 1 to q-1 exactly once (modulo q):

α¹ mod q, α² mod q, α³ mod q, ..., α^(q-1) mod q
produce all distinct values {1, 2, 3, ..., q-1}

Example: For q=19, the primitive roots are: 2, 3, 10, 13, 14, 15

This property is critical — it ensures the shared secret K can take any value from 1 to q-1, giving maximum entropy (unpredictability) in the generated key.

📐 Modular Exponentiation

Computing g^a mod p is called modular exponentiation. It is:

  • Easy to compute in one direction — even for very large numbers, efficient algorithms (square-and-multiply) can compute it quickly
  • Hard to reverse — given only the result and g and p, finding a is the Discrete Logarithm Problem

This asymmetry — easy forward, hard backward — is the one-way function that makes DH secure.

🔑 The Discrete Logarithm Problem

Given: y = g^x mod p

Find: x

This is the Discrete Logarithm Problem (DLP). For large primes (2048+ bits), no known efficient classical algorithm can solve this in a practical amount of time — not even with modern supercomputers.

Eve sees g, p, Y_A, Y_B — but to find X_A or X_B, she must solve the DLP. Computationally infeasible for large p.

🔄 One-Way Function Concept

DirectionOperationDifficultyExample
Forward (Easy)Compute gˣ mod pComputationally easy — polynomial time5⁶ mod 23 = 8 (simple)
Backward (Hard)Find x given gˣ mod pComputationally infeasible for large pFind x where 5ˣ mod 23 = 8 → x = 6 (trivial for small p, infeasible for 2048-bit p)

For practical security: prime p should be at least 2048 bits in length. Research in 2015 showed many deployed DH implementations using only 1024-bit primes were insufficient against well-funded attackers.

This is the clearest way to understand what the algorithm hides and what it reveals. During DH exchange:

Alice Knows
  • p (public prime)
  • g (primitive root)
  • a (her own private key)
  • A = gᵃ mod p (her public key)
  • B (Bob's public key — received)

Bob Knows
  • p (public prime)
  • g (primitive root)
  • b (his own private key)
  • B = gᵇ mod p (his public key)
  • A (Alice's public key — received)
  • s = Aᵇ mod p (shared secret)
  • Does NOT know: a
Eve (Eavesdropper)
  • p (intercepted)
  • g (intercepted)
  • A (intercepted)

  • Does NOT know: b

Security — The Discrete Logarithm Problem

📖 Why DH Is Secure

DH security rests on the Discrete Logarithm Problem (DLP):

  • Computing gˣ mod p given g, x, p → computationally EASY (efficient algorithms exist)
  • Finding x given gˣ mod p, g, p → computationally HARD (no efficient algorithm for large p)

For large primes (600+ decimal digits / 2048+ bits), even the fastest modern computers running the fastest known algorithms cannot solve the DLP in a practical time frame.

✅ Security Best Practices

  • Use prime p of at least 2048 bits (recommended by NIST)
  • Ensure p and q are of appropriate size and satisfy number-theoretic constraints
  • Use ephemeral keys (generate fresh key pairs for each session) → provides Forward Secrecy
  • Combine with authentication (certificates) to prevent MITM attacks
  • Use standardized parameter groups (RFC 7919 — Finite Field DH)

⚠ Known Weaknesses

  • Small primes: If p is too small (like q=353 in the lecture example), brute force is possible
  • Logjam attack (2015): Research showed many deployed DH applications used 1024-bit primes — insufficient against nation-state attackers
  • No authentication: DH itself does not verify identities — vulnerable to MITM (see Section 10)
  • Quantum computers: Shor's algorithm could solve the DLP efficiently — post-quantum cryptography is being developed to address this

⚡ Forward Secrecy — Extra Concept from the Video

Forward secrecy (also called Perfect Forward Secrecy, PFS) means that even if an attacker records all encrypted traffic today and later compromises the long-term private key, they still cannot decrypt past sessions.

DH achieves this by using ephemeral (temporary) keys — fresh DH key pairs are generated for every session and discarded afterwards. Since the session keys are never stored, there is nothing to compromise later.

This is why TLS 1.3 mandates ephemeral DH (DHE or ECDHE) — older TLS versions using static RSA keys did NOT have forward secrecy.

Man-in-the-Middle (MITM) Attack on DH

📖 The Core Vulnerability

DH is a non-authenticated protocol. This means Alice and Bob can establish a shared secret, but neither can prove to the other who they really are. This opens the door to a Man-in-the-Middle (MITM) attack where an active attacker (Mallory) intercepts and replaces the key exchange.

🔴 How the MITM Attack Works

  1. Alice wants to exchange keys with Bob. She sends her public key Y_A.
  2. Mallory intercepts Y_A before it reaches Bob. Mallory creates her own key pair and sends Y_M (her own public key) to Bob pretending to be Alice.
  3. Bob sends his public key Y_B to Alice. Mallory intercepts Y_B and sends Y_M (a different version) to Alice, pretending to be Bob.
  4. Alice computes a shared key K1 with Mallory (thinking it's Bob).
  5. Bob computes a shared key K2 with Mallory (thinking it's Alice).
  6. Mallory can now decrypt, read, and re-encrypt ALL messages between Alice and Bob. Neither Alice nor Bob knows the interception is happening.

Mallory creates TWO separate DH sessions — one with Alice and one with Bob. Both think they're talking to each other.

🛡 Defences Against MITM

  • Digital Certificates (PKI): Alice and Bob present certificates signed by a trusted CA. The certificate binds their public key to their verified identity — Mallory cannot forge this without the CA's private key.
  • Authenticated DH (Station-to-Station protocol): Both parties sign the DH exchange with their long-term private keys, proving identity.
  • Pre-shared Key (PSK): If parties have a pre-shared secret, they can authenticate the DH exchange.
  • Certificate Pinning: Applications hardcode the expected certificate, preventing substitution.

This is why TLS uses DH together with digital certificates — DH handles key exchange, certificates handle authentication.

📖 Where DH Is Used Today

Diffie-Hellman is one of the most widely deployed cryptographic protocols on the internet. It underpins the security of most secure communications you use daily.

🔄 DH Variants

  • Classical DH: Uses multiplicative group of integers modulo p (as covered in this lecture)
  • ECDH (Elliptic Curve DH): Uses elliptic curve groups — much smaller key sizes for the same security level (256-bit ECDH ≈ 3072-bit DH)
  • DHE (Ephemeral DH): Fresh keys for each session → provides Forward Secrecy
  • ECDHE: Combines ECDH + Ephemeral keys — most efficient and secure variant; used in TLS 1.3

🔗 Connection to Other Topics

  • Connects to RSA: RSA also uses public-key cryptography but for encryption and signatures, while DH is for key exchange only
  • Connects to PKI: PKI certificates are used to authenticate DH exchanges, preventing MITM
  • Connects to TLS: TLS certificates + DHE = authenticated + forward-secret HTTPS
  • Historical connection: DH (1976) led directly to RSA (1978) — both built on modular arithmetic hardness
