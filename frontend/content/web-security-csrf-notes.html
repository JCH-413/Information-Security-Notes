01

Cross-Site vs. Single-Site Requests

📖 Overview

This topic covers Cross-Site Requests, the CSRF Attack itself, and the countermeasures used to defend against it.

🏠 Single-Site / Same-Site Request

A request that is sent from a web page to the same website's server (e.g., a facebook.com page sending a request to the facebook.com server). This is the normal, expected behavior.

🔀 Cross-Site Request

Suppose a user opens two different web pages of different websites: 1) www.facebook.com and 2) www.web.com. The user is logged in to Facebook, and a session ID is created.

  • The web.com page contains a form of facebook.com.
  • As you fill in the Facebook form on web.com, the request will be sent to the Facebook server.

02

Cross-Site Request Vulnerabilities

⚠️ First Vulnerability — Automatic Cookie Attachment

As the user sends a request to the Facebook server through web.com, the cookies of the Facebook web page are automatically attached with the request.

  • This is not hijacking — it is the natural behavior of the browser.
  • But web.com can copy the session easily. So this is a vulnerability of cross-site requests.

⚠️ Second Vulnerability — Unverified Origin

Suppose a request is sent through web.com to the Facebook server. If the Facebook server doesn't know where the request came from, it will respond to the web.com page with user data. This is also a vulnerability.

🍪 Session Cookies

There is a type of cookie called a "Session cookie" that can attach to both sites. That may cause the problem.

🎯 Session Hijacking

  • In TCP, if you get the sequence number (seq #), you can hijack the session.
  • If an attacker puts their web page on the victim's computer, the request will go from A to B.
  • Meanwhile, if B can't recognize the difference, then it's a problem.

📡 CSRF Attack on IoT Devices

IoT devices don't have countermeasures to deal with CSRF attacks, and they use a web server for their working — making them vulnerable.

03

Launching a CSRF Attack — GET vs. POST

🚀 How to Launch a CSRF Attack

  • The attacker must know the method of the HTTP request before launching the attack.
  • The method can be GET or POST, which can be analyzed through the "HTTP Header Live" extension in Firefox, or Wireshark.
AspectGETPOST
Data locationIn the header / URL portionIn the body portion
VisibilityUser can see data in the URLData hidden in request body
In the GET method, data is attached in the header of the HTTP request and the user can see it in the URL. In the POST method, data is attached in the body part of the HTTP request.
04

CSRF Attack on GET Services

🎯 Objective

The objective of this attack is to add the attacker as a friend in the victim's friend list — i.e., add yourself to someone's friend-list without their consent.

🔍 Investigate the GET Method

The attacker investigates the GET method request before launching the attack:

  • When Alice clicks "add friend", we can observe the GET request.
  • Goal is to capture that traffic — note the value of "add friend".
  • Disable the countermeasures if they are attached (doesn't matter).
  • All we need to know is the friend's ID.
  • The randomly generated session ID is granted after giving login credentials — and by design the browser adds it for us.

🛠️ Preparation of Forged GET Request

  • The attacker prepares a forged GET request in a web page with their own ID.
  • As the victim clicks on the page link, the forged GET request will trigger.
  • You have to play with their minds — once the user clicks this request, the friend request will be sent.
  • The victim needs to have an active session with the social media site.

⚡ Forge the GET Request — Why It's Easy

  • The browser triggers a GET request as soon as it sees an <img> or <iframe> tag in the HTML.
  • An attack on a GET HTTP request is easy, because the data is attached in the header of the HTTP request or in the URL.

⚙️ Mechanism (Forge the GET Request)

  • Suppose Alice and Samy (Attacker) are two users on a social network like Facebook.
  • Samy wants to add Alice to his friend list without Alice's permission.
  • Alice is logged into her account and her session ID is created.
  • Samy sends a URL of a web page to Alice that contains a forged GET request with Samy's profile ID.
  • As Alice clicks the link, the browser triggers the HTTP GET request. Alice's session ID is attached with the request automatically.

05

CSRF Attack on POST Services

🎯 Objective

The objective of this attack is to edit the victim's profile.

🔍 Investigate the POST Method

  • The attacker investigates the POST method request before launching the attack.
  • The POST request will be sent in forms, because the objective of the attack is to edit the victim's profile.

🛠️ Preparation of Forged POST Request

The attacker prepares a forged POST request in a web page with the victim's ID. As the victim clicks on the page link, the forged POST request will trigger.

⚙️ Mechanism (Forge the POST Request)

  • Suppose Alice and Samy (Attacker) are two users on a social network like Facebook.
  • Samy wants to edit Alice's profile without Alice's permission.
  • Alice is logged into her account and her session ID is created.
  • Samy sends a URL of a web page to Alice that contains a forged POST request with Alice's profile ID.
  • As Alice clicks the link, the browser triggers the HTTP POST request. The session ID attaches automatically.
06

Countermeasures: Fundamental Cause & Referral Header

⚠️ Fundamental Cause

The browser knows whether an HTTP request is single-site or cross-site, but the server doesn't know. This is the vulnerability, because the protected shield must be deployed on the server side, not at the browser side.

📋 Referral Header

  • The referral header tells the server where the request came from.
  • But the referral header carries a lot of user information that can harm user privacy. So many users block the referral header via a proxy.

Therefore, the referral header is not a reliable solution.

07

Countermeasure: Secret Token

🔐 Secret Token

  • The server attaches a secret token with each web page to know whether a request is single-site or cross-site.
  • Tokens are strings of special characters.
  • The property of the token is hidden so that a malicious user cannot steal it.

⚙️ Generating and Using the Token

The developer can generate the token inside the page dynamically, and then send the HTTP request with the token attached.

📡 Token Approach in IoT

The same secret-token approach can be applied to IoT devices to defend them against CSRF attacks.

08

Countermeasure: Same-Site Cookies

⚠️ The Problem

As a request is sent to the Facebook server from a web page of web.com, the cookies of facebook.com are attached automatically. The problem is that the Facebook server can't recognize whether the request comes from a Facebook web page or a web.com web page.

🍪 Setting Cookies and Their Types

  • Simple cookies — attached with cross-site requests.
  • Lax cookies — sometimes attached with cross-site requests and sometimes not (depends on the developer).
  • Strict cookies — only attached with same-site requests.
Cookie TypeAttached to Cross-Site Request?
SimpleYes — always attached
LaxSometimes (depends on developer)
StrictNo — same-site only
Setting cookies to Strict blocks them from cross-site requests, directly defeating the automatic cookie attachment that makes CSRF possible.

📝 Common Exam Questions
  • What is a cross-site request and how does it differ from a single-site request? → A cross-site request goes from one website's page to another website's server, vs. same-site requests staying within one site
  • Why is automatic cookie attachment a vulnerability? → The browser attaches the target site's cookies automatically, letting a malicious site send authenticated requests / copy the session
  • What is the difference between GET and POST in terms of data placement? → GET puts data in the header/URL; POST puts data in the body
  • Why is forging a GET request easy? → The browser auto-triggers GET on <img>/<iframe> tags, and data sits in the URL/header
  • Describe the mechanism of a CSRF GET attack. → Samy sends Alice a page with a forged GET request carrying Samy's ID; Alice's session ID attaches automatically on click, adding Samy as a friend
  • What is the objective of a CSRF POST attack? → To edit the victim's profile
  • What is the fundamental cause of CSRF? → The browser knows the request origin but the server does not; protection must be on the server side
  • Why is the referral header not a reliable countermeasure? → It leaks user privacy info, so many users block it via proxy
  • How does a secret token defend against CSRF? → The server embeds a hidden token per page; cross-site requests lack the valid token and are rejected
  • List the three same-site cookie types and their behavior. → Simple (always attached cross-site), Lax (sometimes), Strict (same-site only)
