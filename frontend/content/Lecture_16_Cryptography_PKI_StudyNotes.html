PKI Basics — Foundations of Public Key Cryptography

📖 Historical Background

All early cryptographic systems (including Rotor Machines used in World War II) were based on permutation and substitution of characters. As computers became available, more complex symmetric systems like DES and AES were developed — but all still relied on permutation and substitution.

Public-key cryptography was fundamentally different: it is based on mathematical functions rather than permutation and substitution. The idea was proposed in 1976 by Diffie and Hellman — a landmark moment in cryptography history.

💡 Why Was Public-Key Cryptography Needed?

Public-key encryption was developed to solve two critical problems that symmetric cryptography could not address:

  1. Key Distribution Problem — How to establish secure communication without needing to first share a secret key through a secure channel (or trust a Key Distribution Centre with your key)
  2. Digital Signatures Problem — How to verify that a message truly comes intact from the claimed sender (authenticity + integrity)

⚠ Common Misconceptions About Public-Key Crypto

  • Misconception 1: "Public-key algorithms are completely secure" — No algorithm is unconditionally secure; security is computational (infeasibility-based)
  • Misconception 2: "Public-key encryption makes symmetric encryption obsolete" — False. Symmetric encryption is much faster; asymmetric is used for key exchange and signatures, not bulk data
  • Misconception 3: "Key distribution is solved completely" — Key distribution is easier but the trust problem (how do you know a public key really belongs to who you think?) led to the need for PKI itself

🏗 PKI — Formal Definition (from PDF)

Public Key Infrastructure (PKI) is a framework of Technologies, Policies, and Procedures that enables secure communication and data exchange using public key cryptography. It provides a reliable way to:

  • Verify the identity of users, devices, or services
  • Protect data through encryption
  • Authenticate through digital signatures

🔑 Characteristics of Public-Key Encryption (Technical)

Public-key algorithms rely on two keys with these mathematical characteristics:

  • It is computationally infeasible to find the decryption key knowing only the algorithm and the encryption key
  • It is computationally easy to encrypt or decrypt messages when the relevant key is known
  • Either of the two related keys can be used for encryption — with the other used for decryption (in some schemes like digital signatures)

The Diffie-Hellman (1976) proposal: each user generates a key pair; one key is placed in a "public ring" (public directory) and the other is the private key.

Public Key Cryptography (Asymmetric Encryption)

📖 Definition

A cryptographic system where every user has two mathematically linked keys — a public key (shared openly) and a private key (kept secret). Also called asymmetric encryption because two different keys are used for encryption and decryption.

🔑 Key Pair Properties

  • Public and private keys are mathematically linked
  • Cannot guess or derive one key from the other
  • Only the matching key from the pair can decrypt
  • Public key → shared on internet, social media, email
  • Private key → confidential, stored securely

Example from slides: Sundar encrypts with Venkatesh's Public Key → Venkatesh decrypts with his own Private Key. No other key can decrypt the message.

🧠 Beginner Explanation

Think of the public key like a padlock you give to everyone. Anyone can lock a box using your padlock. But only YOU have the key to open it (private key). No one else can open the box even though everyone has your padlock.

🏛 Technical Explanation

Asymmetric cryptography uses a trapdoor one-way function (e.g., RSA uses prime factorization difficulty, ECC uses elliptic curve discrete logarithm). Encrypting with public key guarantees only the private key holder can decrypt — mathematically infeasible to reverse without the private key.

⚠ Common Mistakes Students Make

  • Confusing public and private key roles — remember: public key encrypts, private key decrypts (for confidentiality)
  • Thinking symmetric = two keys. It's the opposite — symmetric uses ONE shared key
  • Assuming asymmetric is faster than symmetric — it is actually SLOWER (used for key exchange, not bulk data)

Hashing / Hash Functions

📖 Definition

A hash function is a mathematical algorithm that converts any input data of arbitrary length into a fixed-length hexadecimal number. No keys are required. The output is called a Hash Value or Message Digest. It is also called one-way encryption because you cannot reverse-engineer the original message from the hash.

✅ Key Properties

  • Fixed-length output regardless of input size
  • Deterministic: Same input → always same hash
  • Avalanche Effect: Even 1 character change = completely different hash
  • One-way: Cannot retrieve original message from hash
  • No keys required (unlike encryption)
  • Also called the digital thumbprint of data

📊 SHA-256 Example (from slides)

Hello Worlda591a6d40bf42040…
Hello World!7f83b1657ff1fc53…
Cryptography is awesome2d601088ecb12661…

SHA-256 always outputs 256 bits = 32 bytes = 64 hex characters. Note how adding just "!" completely changes the hash — Avalanche Effect.

Common algorithms: MD5 (128-bit, deprecated), SHA-1 (160-bit, deprecated), SHA-256 (256-bit), SHA-512 (512-bit)

🧠 Beginner Explanation

A hash is like a fingerprint of data. Just as every person has a unique fingerprint, every piece of data has a unique hash. If you change even one letter in a document, the fingerprint changes completely. And you can't reconstruct a person from their fingerprint — that's why it's one-way.

🌍 Real-World Use

  • Password storage (systems store hash of password, not password itself)
  • File integrity checking (download verification)
  • Digital signatures (sign the hash, not the whole document)
  • Blockchain (each block linked via hash of previous block)

Public Key Infrastructure (PKI)

📖 Definition

PKI is a collection of hardware, software, policies, procedures, people, and roles that helps to create, manage, distribute, use, store, verify, and revoke digital certificates — and manages their complete lifecycle.

PKI creates a Digital Certificate that contains the link between a public key and the person/entity to whom the certificate was issued.

🎯 Purpose of PKI

  • Simplify security of online transactions (email, internet banking)
  • Identify parties involved in a transaction and verify integrity of information shared
  • Used when password-based authentication is not sufficient — requires stronger authentication
  • Provides the 4 Trust Services: Authentication, Confidentiality, Integrity, Non-Repudiation

Chain of Trust & Trust Store

📖 Chain of Trust — Definition

The Chain of Trust (also called Certificate Chain) is a hierarchical structure of digital certificates that links an end-entity certificate back to a trusted Root CA. It establishes trust by verifying each certificate is signed by the authority above it.

🏗 Structure of the Chain of Trust

🏛 Root CA
Trust Anchor · Self-Signed · Highest Trust
→ Signs Intermediate CA's certificate
↓
🔗 Intermediate CA
Layer of protection · Can have sub-CAs
→ Signs End Entity certificate
↓
📄 End Entity Certificate
Issued to person/server/org/website

Key insight: Root CA is self-signing — it signs its own certificate. The Intermediate CA acts as a buffer, so Root CA's private key is rarely used (kept offline for security).

🔍 Certificate Verification Process — Step by Step

  1. Customer presents their certificate (usually includes full chain to Root CA)
  2. Verifying entity uses the issuing authority's public key to verify the certificate signature
  3. If the verifying entity trusts this CA → verification complete. Stop.
  4. If NOT trusted → move up the chain to the next CA and repeat verification
  5. Continue until a trusted CA is found or you reach the Root CA
  6. If Root CA is not in trust store → certificate is rejected

🏪 Trust Store

A Trust Store (also: Certification Store) is a collection of Root CAs that are trusted by default. It is maintained by the organizations that create operating systems and browsers.

Digital Signatures & Certificate-Based Authentication

📖 Digital Signature — Definition

A digital signature is a cryptographic mechanism used to verify the authenticity and integrity of a digital document or message. It proves: (1) who sent it (authentication), and (2) it was not tampered with (integrity), and (3) the sender cannot deny it (non-repudiation).

🔒 Certificate-Based Authentication — 4 Validation Questions

Before granting access, the system asks 4 questions. Failure on ANY ONE = access denied.

  1. Trusted CA? — Was the certificate issued by a trusted CA? Is the intermediate and Root CA trusted?
  2. Validity period? — What is the issue and expiry date? Is the certificate expired?
  3. Revocation status? — Has the certificate been revoked for any reason? (Checked via CRL or OCSP)
  4. Proof of ownership? — Can the user prove they have the private key associated with this certificate?

💳 Smart Cards

  • Compliant with ISO/IEC 7810 ID-1, 7816 & 14443
  • Can perform cryptographic operations within the card itself: encryption, decryption, digital signatures, hashing, key pair creation
  • Secure storage for private keys, digital certificates, account numbers, passwords
  • Only public key and digital certificates are exposed — private keys stay inside
  • Use cases: National ID (Aadhaar), E-Passport, Banking (Debit/Credit), Smart SIM, Building access

Code Signing Certificates

📖 Definition

Code Signing is the process of digitally signing software (executables, binaries, scripts, source code) to identify the publisher and assure that the software has not been modified or tampered with since it was signed.

✅ What Code Signing Proves

  • Identity of the software publisher (who made it)
  • Software has not been tampered with after signing (integrity via hash values)
  • The publisher's certificate was issued by a CA connected to a trusted Root CA (chain of trust)

Code signing does NOT prove the software is safe or malware-free — only that it came from a verified publisher unchanged.

🔑 Security Dependency

The entire system's security depends on the software publisher keeping the private key confidential. If the private key is compromised, attackers can sign malicious software with the publisher's identity.

Best practice: Store private keys in a Hardware Security Module (HSM) — a tamper-resistant cryptographic hardware device.

Time Stamping

📖 Definition

Time Stamping is the process of securely recording the creation or modification time of a document. The output is called a Trusted Time Stamp. Once timestamped, the document cannot be modified — even by the document's owner.

✅ Benefits of Timestamping

  • Increases integrity of digital signatures and code signing
  • Provides secure record of date and time of transaction
  • Key benefit: A digital signature WITHOUT timestamp expires when the underlying certificate expires. WITH timestamp → signature remains valid even after certificate expiry
  • Can timestamp separately or while signing

⚙ How It Works

  1. Document's Hash Value is sent to Time Stamping Authority (TSA)
  2. TSA combines Hash + Current Time
  3. TSA encrypts combined value with TSA's Private Key → creates Time Stamp Token
  4. Time Stamp Token attached to document
  5. TSA syncs time with Authoritative Time Server (NTP)

In India: Authoritative time server is managed by the National Physics Laboratory. All CAs and TSAs in India must sync with it (required by CCA).

Critical exam point: A digital signature expires with its certificate. Adding a timestamp makes the signature permanently valid beyond certificate expiry.

  • MCQ: What happens to a digital signature without a timestamp when the certificate expires? → The signature becomes invalid
  • Short: What is a Time Stamping Authority (TSA)? What protocol does it use to sync time? → NTP
