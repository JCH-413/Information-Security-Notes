01

User Authentication & the NIST Model

📖 What is User Authentication?

User authentication is a two-step process by which a system confirms you are who you claim to be, so it can then set your permissions and track your activity.

  • Identification — You claim an identity by presenting a username / User ID (e.g. walking up and saying "Hi, I'm Alice"). This part is usually not secret.
  • Verification — You prove the claim by providing a secret (your password). The system checks this secret against what it has on file.

In the worked example: "ABTOKLAS" is Alice's public identifier, and her password is the secret authentication information. If both match, the system trusts the claim.

🏛️ NIST Model for Electronic User Authentication

A standard framework defined by the National Institute of Standards and Technology (NIST), per the SP 800-63-2 Architectural Model. It involves several distinct actors.

Registration Authority (RA)

The trusted office that checks your physical ID (like a passport) and vouches for you. Performs the initial identity proofing.

Credential Service Provider (CSP)

The entity that creates and issues your electronic credential — like a digital ID card. In many systems the RA and CSP are part of the same organization.

Subscriber / Claimant

The user being authenticated.

Verifier

When you later try to log in, the verifier is the software that challenges you for your password or token.

Relying Party (RP)

The final application or service you want to access (e.g. your email server). It trusts the Verifier's decision.

02

Means of Authentication (Factors & Methods)

📖 The Four Factors of Authentication

You can prove your identity in four general categories, called factors of authentication:

Something you know

The most common factor — a password, a PIN, or answers to security questions. (Knowledge-based)

Something you possess

A physical object you have — a smart card, a security token that generates codes, or even your physical house key. (Possession-based)

Something you are (Static Biometrics)

Your unique physical characteristics — fingerprint, retina pattern, or face. (Inherence-based)

Something you do (Dynamic Biometrics)

Your behavioral patterns — voice rhythm, handwriting, or unique typing rhythm.

Multi-factor authentication (MFA) combines these for stronger security. For logging into a network, the most common combination is something you know (a password) used with something you possess (a cryptographic key).

🔧 Common Authentication Methods

  • Passwords — the most widely used method, but vulnerable to brute force, phishing, and dictionary attacks. Best practice: strong unique passwords stored securely (e.g. hashing with salt).
  • Biometrics — fingerprints, voice recognition, facial scans. High security but raises privacy concerns and potential risk of spoofing.
  • Two-Factor Authentication (2FA) — adds an extra layer with a secondary step such as a one-time password (OTP) sent to a device.
  • Public Key Infrastructure (PKI) — uses cryptographic techniques to authenticate entities, ensuring secure communication.

🚀 Emerging Trends

  • Passwordless Authentication — relies on biometrics and security keys, reducing dependency on traditional passwords.
  • Behavioral Biometrics — analyzes patterns like typing speed and mouse movements for continuous authentication.
  • Decentralized Authentication — blockchain-based systems that eliminate central authorities, reducing risk of breaches.

⚠️ Challenges in Authentication

  • Credential Theft — stolen credentials via phishing or social engineering pose significant risks.
  • Usability vs. Security — balancing user convenience with robust security is hard.
  • Scalability — adapting systems for large-scale environments with diverse users and devices.

✅ Best Practices for Authentication

  • Employ MFA wherever possible.
  • Regularly audit and update authentication policies.
  • Educate users on recognizing and avoiding phishing attempts.
  • Use adaptive authentication systems that assess context and risk level (e.g. unusual locations or devices).

Authentication is not a one-size-fits-all solution but a continuously evolving component of information security, adapting to new threats and technologies.

Mutual & One-Way Authentication

📖 Mutual Authentication

Often it is not enough for just the server to verify you — you also want to verify you are talking to the real server and not a fake one set up by a hacker. This two-way verification is called mutual authentication.

The main goals are to (1) confirm each other's identity and (2) securely exchange a session key to encrypt the rest of their conversation.

⚠️ Biggest Threat: Replay Attacks

An eavesdropper records your login sequence and plays it back later to impersonate you. There are different types:

  • Type 1 — simply copy and replay the message later.
  • Type 2 & 3 — replay a message while its timestamp is still valid, possibly while blocking the original message.
  • Type 4 — send your own message back to you, which can cause confusion (reflection).

🛡️ Countering Replay Attacks — Two Weapons

1. Timestamps

Every message gets a 'born-on' date. If a message arrives too late, it is rejected.

Problem: requires all computers to have perfectly synchronized clocks. If clocks drift apart, it causes big problems.

2. Challenge / Response

Like a secret handshake. Very secure, but requires extra back-and-forth messages (a handshake), which isn't ideal for all situations.

🔁 Challenge/Response — Worked Example

Step 1 — Issue the challenge

You (Party A) say: "Prove you're real by answering this random number I just made up: 57."

Step 2 — Respond

The server (Party B) takes that number, encrypts it with your shared secret key, and sends the encrypted result back.

Step 3 — Verify

You decrypt it; if it gives back '57', you know it's the real server.

→ Identity confirmed without exposing the secret key

📧 One-Way Authentication

Not all authentication needs to be mutual. The classic example is email — sender and recipient aren't online at the same time. This creates special requirements:

  • The envelope (the 'To:' and 'From:' addresses) must be readable by the email system so it can route the message.
  • The body of the message should be encrypted so mail servers can't read it.
  • The recipient wants assurance that the email actually came from you.

So we need a way to authenticate the sender without requiring the recipient to be online for a complex handshake.

04

Remote Authentication Using Symmetric Encryption

📖 The Key Distribution Center (KDC)

With symmetric encryption everyone shares secret keys, but managing a secret key between every possible pair of users is a nightmare. The solution is a trusted middleman called a Key Distribution Center (KDC).

  • You and every other user have a single, long-term master key shared only with the KDC.
  • When you want to talk to someone else, you ask the KDC for a temporary session key.
  • The KDC generates this session key and sends it to both parties, encrypted inside a package using each one's respective master key.

Benefit: you only have to protect one master key, and the KDC handles the rest.

📧 One-Way Authentication with a KDC (email)

The protocol is similar to the first part of Needham-Schroeder, but with the mutual handshake removed. Alice gets a ticket for Bob and the session key from the KDC, then sends Bob two things:

  • The ticket (which Bob can decrypt with his master key).
  • The actual email message, encrypted with the session key.

Bob uses his master key to open the ticket, gets the session key, and uses it to decrypt the email. This proves the message came from someone who got a valid ticket from the KDC (i.e. Alice). Not as strong as mutual authentication, but it works for email.

⚠️ Suppress-Replay Attack (the catch with timestamps)

Timestamps introduce their own clever attack. It exploits clock drift between two parties.

The Scenario

Imagine Alice's clock is 5 minutes ahead of Bob's.

Step 1

Alice sends a request with her timestamp — which is 5 minutes in Bob's future.

Step 2

An attacker, Eve, intercepts and holds onto this message.

Step 3

5 minutes later, Bob's clock 'catches up' to the timestamp. Eve now replays the message.

→ Bob sees a fresh, valid timestamp and accepts the replayed message!

Solutions: either constantly synchronize clocks very carefully, or go back to using nonces (random challenges), which are not vulnerable to this timing trick.

Kerberos

📖 What is Kerberos?

  • A trusted third-party authentication service.
  • Developed at MIT as part of Project Athena.
  • Relies exclusively on symmetric encryption (no public-key crypto).
  • Provides secure mutual authentication between users and servers in a network.
  • Uses a protocol based on Needham-Schroeder with enhancements to prevent replay attacks.

📋 Kerberos Requirements (Secure-Reliable-Transparent-Scalable)

  • Secure — a network eavesdropper cannot obtain information to impersonate a user.
  • Reliable — highly available using a distributed server architecture.
  • Transparent — users are unaware of authentication beyond entering a password.
  • Scalable — supports large numbers of clients and servers with a modular design.

🧩 Key Components

Authentication Server (AS)

  • Verifies user identity at login.
  • Issues Ticket-Granting Tickets (TGTs).

Ticket-Granting Server (TGS)

  • Issues service-specific tickets.
  • Acts as a trusted intermediary.

Ticket

  • A reusable credential proving the user's identity to a service.
  • Encrypted with the server's secret key.

Authenticator

  • A one-time-use credential proving the user is the ticket owner.
  • Encrypted with a session key.

🏰 The Kerberos Realm

A Kerberos realm has three groups of parties, with the Key Distribution Center (KDC) in the middle housing the AS and TGS:

  • Clients / Users (left)
  • Key Distribution Center = Authentication Server + Ticket Granting Server (centre)
  • Services (right)

The two message types that flow are Authenticators (one-time proofs) and Tickets (reusable credentials).

🔄 The Kerberos Message Flow (6 messages)

The protocol proceeds through three exchanges — with the AS, then the TGS, then the Service:

Step 1

User → Authentication Server

The user requests authentication. The message attributes include: User Name/ID (e.g. "Rob"), Service Name/ID (e.g. "CRM"), User IP address, and the Requested lifetime for the TGT. The AS holds a table of each user's Name/ID and Client Secret Key.

Step 2

Authentication Server → User (issues the TGT)

The AS returns a session-key block (attributes: TGS Name/ID, Timestamp, Lifetime — same as TGT) and the Ticket Granting Ticket (TGT).

TGT attributes: User Name/ID • TGS Name/ID • Timestamp • User IP address • Lifetime for TGT
Step 3

User → Ticket Granting Server

The user presents the TGT to the TGS to request access to a specific service.

Step 4

Ticket Granting Server → User (issues the Service Ticket)

The TGS validates the TGT and returns a Service Ticket (ST).

Step 5

User → Service

The user presents the Service Ticket (ST) to the target service.

Step 6

Service → User

The service confirms and grants access — completing mutual authentication.

Analogy — The Ticket Booth

Think of the AS as the front desk that checks your ID once and hands you a reusable day-pass (TGT). Whenever you want to enter a specific ride (a service), you show the day-pass at the ride-ticket booth (TGS), which gives you a single-ride ticket (ST). The ride operator (service) only needs to check that single-ride ticket — you never expose your original password again.

📝 Common Exam Questions
  • Where was Kerberos developed and what crypto does it use? → MIT (Project Athena); symmetric encryption only.
  • Difference between a Ticket and an Authenticator? → Ticket = reusable, encrypted with server's secret key; Authenticator = one-time-use, encrypted with session key.
  • What does the AS issue vs. the TGS? → AS issues the TGT; TGS issues service-specific tickets (ST).
  • List the four Kerberos requirements. → Secure, Reliable, Transparent, Scalable.

Access Control Fundamentals

📖 Authentication vs. Access Control

Authentication verifies identity using credentials like passwords, biometrics, or tokens. Authorization (Access Control) determines what resources a user can access and grants access based on that level.

Example: a user might have access to a restaurant (authenticated), but there are specific things they are not allowed to do (authorization limits).

🔑 The Four Stages: Identification → Authentication → Authorization → Access

AuthorizationPermission granted for admittanceMia opens door to allow delivery person inUser authorized to log in
AccessRight given to access specific resourcesDelivery person can only retrieve box by doorUser allowed to access only specific data

🧱 Authentication Building Blocks (recap)

  • Authentication Factors: Knowledge-based (passwords, PINs), Possession-based (smart cards, tokens), Inherence-based (fingerprints, retina scans).
  • Multifactor Authentication (MFA): combines two or more factors (e.g. password + smartphone app).
  • Single Sign-On (SSO): one login for multiple systems — reduces password fatigue but increases risk if compromised.
  • Authentication Protocols: examples include Kerberos, OAuth, and OpenID Connect, securing authentication during data exchange.

Access Control Models (DAC, MAC, RBAC, ABAC)

📖 The Four Models

  1. Discretionary Access Control (DAC)
  2. Mandatory Access Control (MAC)
  3. Role-Based Access Control (RBAC)
  4. Attribute-Based Access Control (ABAC)

1️⃣ Discretionary Access Control (DAC)

DAC empowers the owners of resources to decide who can access them. Control over permissions lies entirely with the resource owner.

Core characteristics:

  • Ownership-centric — the owner holds complete authority over access rights.
  • Highly customizable — permissions tailored for specific users or groups.
  • Access Control Lists (ACLs) — lists defining which users/groups can read, write, or execute the resource.

DAC — Benefits

  • Straightforward and easy for users to manage.
  • Provides fine-grained control over who can do what.

DAC — Drawbacks

  • Susceptible to privilege escalation and insider threats.
  • Absence of centralized oversight complicates enforcement of organization-wide policies.

Illustration: When a user creates a file on their device, they can specify exactly which other users may read, modify, or run that file. (Diagram: Owner → specifies users/groups who can access → Object.)

2️⃣ Mandatory Access Control (MAC)

MAC is a model where access rules are strictly enforced by a central authority, and users have no ability to alter these rules.

Core characteristics:

  • Security Labels & Clearance Levels — both users and resources receive classifications (e.g. "Top Secret", "Confidential"); access is permitted only when labels align.
  • Non-Discretionary Access — users cannot modify permissions themselves.
  • Centralized Authority — only system administrators can set or change access policies.
Privilege levels (high → low): Top Secret → Secret → Confidential → Public

MAC — Benefits

  • Delivers a high level of security.
  • Perfectly suited for settings demanding rigorous confidentiality — military or government operations.

MAC — Drawbacks

  • Can be rigid and less intuitive for users.
  • Challenging to deploy and maintain where frequent changes are needed.

Illustration: A document marked "Top Secret" is accessible exclusively to individuals holding "Top Secret" clearance.

3️⃣ Role-Based Access Control (RBAC)

RBAC manages permissions by assigning them to roles instead of individual users, making access control more efficient.

Core characteristics:

  • Role-focused — users receive roles such as Admin, Editor, or Viewer, each with predefined access rights.
  • Highly scalable — ideal for complex hierarchies and large user bases.
  • Duty Segregation — roles structured to avoid conflicts of interest (e.g. one person cannot both approve and audit the same transaction).

Diagram flow: Admin assigns users to roles → Users assigned to roles → Roles define authority level → Permissions authorized by role.

RBAC — Benefits

  • Simplifies administration in large enterprises.
  • Enhances ease of auditing and meeting compliance requirements.

RBAC — Limitations

  • Setting up roles initially can be intricate and time-consuming.
  • Lacks flexibility to adapt to contextual or real-time conditions.

Practical example: An "HR Manager" role gets access to employee information, whereas a "Finance Manager" role would have permissions related to payroll data.

4️⃣ Attribute-Based Access Control (ABAC)

ABAC determines access by evaluating attributes of the user, the resource, and the environment — enabling real-time, adaptive access management.

Core characteristics:

  • Policy-driven — permissions governed by rules that assess specific attributes.
  • Context-sensitive & dynamic — role, current time, location, and device type influence decisions.
  • Granular access — supports detailed, conditional policies for precise control.

Diagram attribute groups: User attributes (Name, Role, Security clearance) • Resource attributes (Creation date, Resource owner, Data sensitivity) • Environmental attributes (Access time, Data location, Threat levels).

ABAC — Benefits

  • Exceptional flexibility — suited for frequently changing environments.
  • Capable of handling intricate policies involving multiple attribute combinations.

ABAC — Challenges

  • Can impose significant processing demands.
  • Crafting and maintaining policies can be complex and resource-intensive.

Illustrative example: Access permitted only if the individual is an employee (attribute: role), accessing during business hours (attribute: time), and using a company-approved device (attribute: device type).

DoS & DDoS Attacks

📖 Denial of Service (DoS)

A network security attack in which the hacker makes the system or data unavailable to someone who needs it. The hacker tries to make a network, system, or machine unavailable by flooding it with fake requests or traffic. This prevents real users from accessing it — causing anything from slowdowns to complete shutdowns.

🌐 Distributed Denial of Service (DDoS)

DDoS attacks use multiple systems, often compromised computers (botnets), to attack a single target. Two examples are amplification attacks and botnet-based attacks:

  • Amplification attack — attackers use services like DNS to send a small query that generates a large response, flooding the victim with data.
  • Botnet-based attack — coordinates many infected computers to send attack traffic from multiple sources, making it hard to defend against.

Operation of a DDoS attack: attacker computers send malicious traffic → through the Internet → to the target server, while real users' clean traffic is crowded out → target goes "Out of Resources / Service Offline".

🗂️ Types of DoS / DDoS Attacks

Volume-Based Attacks

Floods the network with too much data.

Protocol Attacks

Exploit weaknesses in network protocols.

Application Layer Attacks

Make target applications crash or become sluggish.

DDoS Attacks

Use multiple systems to attack a single target.

Resource Exhaustion

Repeatedly request access to overload the application.

Reflective Attacks

Send requests to 3rd-party servers using the victim's IP address (responses bounce back to the victim).

Wireless Security

⚠️ Why Wireless Security is Critical — Key Risk Factors

  • Channel — broadcast nature → prone to eavesdropping and jamming.
  • Mobility — portability increases risk of loss / theft / unauthorized access.
  • Resources — limited memory/processing in mobile devices → vulnerable to malware.
  • Accessibility — devices left unattended → physical attacks.

🎯 Wireless Network Threats

  • Accidental Association — e.g. an employee's laptop connects to a neighbouring company's Wi-Fi unintentionally.
  • Malicious Association — e.g. an attacker sets up a fake "Free Airport Wi-Fi" AP to capture login credentials.
  • Ad Hoc Networks — e.g. two laptops connected directly without an AP → no central control.

🧩 Wireless Components & Attack Points

  • Wireless Client — phone, laptop, IoT device.
  • Access Point — Wi-Fi hotspot, cell tower.
  • Transmission Medium — radio waves (vulnerable to interception).

Example: a Wi-Fi-enabled laptop in a café connects to a rogue AP instead of the legitimate one.

🔒 Securing Wireless Transmissions

Threat: unauthorized network access.

Solution: IEEE 802.1X Port-Based Access Control — requires authentication before granting network access; prevents rogue APs.

Example: an employee must enter username / password / certificate to connect to corporate Wi-Fi. Also: encrypt all Wi-Fi traffic so intercepted data is unreadable.

📡 Securing Wireless Access Points

Goal: prevent eavesdropping, tampering, disruption.

Signal hiding techniques: disable SSID broadcasting, use cryptic SSID names, reduce signal strength, use directional antennas.

Encryption: use WPA3, AES-CCMP.

✅ Securing Wireless Networks — Best Practices

  • Use encryption (WPA3 for router-to-router).
  • Install antivirus / firewall on all endpoints.
  • Turn off identifier broadcasting.
  • Change default router SSID.
  • Change default admin password.
  • Use MAC filtering — with caution, as it can be spoofed.

Example: Home Wi-Fi → change SSID from "Linksys" to "HomeNet_secure", disable SSID broadcast, enable WPA3.

Firewalls

📖 What is a Firewall?

A firewall is a network security system that:

  • Creates a controlled barrier between trusted internal networks and untrusted external networks (e.g. the Internet).
  • Inspects and controls traffic based on security rules.
  • Provides a single point of monitoring and enforcement.

"The function of a strong position is to make the forces holding it practically unassailable." — Carl von Clausewitz.

Example: a company uses a firewall to allow employees to browse the web while blocking hackers from entering the network.

🧱 Firewall Characteristics

Firewalls are designed to:

  • Force all traffic through them — no bypass allowed.
  • Permit only authorized traffic — defined by security policy.
  • Be immune to attacks — run on hardened, secure systems.

Control techniques:

  • Service Control — what services (HTTP, SMTP) are allowed.
  • Direction Control — which way traffic can flow.
  • User Control — who can access (via authentication).
  • Behavior Control — how services are used (e.g. block spam).

Example: only HR can access the payroll server, and only during work hours.

🗂️ Types of Firewall

Packet Filtering Firewall

Filters based on IP, port, protocol. Fast but minimal inspection.

Example: block all traffic from a specific IP.

Stateful Inspection Firewall

Tracks active connections in a state table; only allows return traffic for established sessions.

Example: allow an inbound response only if an outbound request was made first.

Application-Level Gateway (Proxy)

Acts as an intermediary for app traffic (e.g. HTTP, FTP); inspects full content and enforces detailed policies.

Example: a web proxy that blocks malicious sites.

Circuit-Level Gateway

Relays TCP connections without deep inspection; often used with the SOCKS protocol.

Example: a secure relay for remote-user VPN access.
