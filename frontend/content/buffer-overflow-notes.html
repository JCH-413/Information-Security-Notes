02

What is a Buffer Overflow?

📖 Definition

A buffer overflow (or buffer overrun) occurs when more data is put into a fixed-length buffer than the buffer can handle. The extra information, which has to go somewhere, can overflow into adjacent memory space, corrupting or overwriting the data held in that space.

It is a common software coding mistake that an attacker could exploit to gain access to your system.

  • A buffer is sequential memory.
  • Buffer overflow is a common and classic vulnerability — one of the oldest in software.
Analogy

Think of a buffer like a glass sized for exactly 12 ounces. Pour in 30 ounces and the extra spills onto whatever is sitting next to the glass — in memory, that "whatever" can be critical control data like the return address.

Virtual vs. Physical Address Scheme

🧠 How the OS Allocates Memory

  • For every process, the OS allocates one or several blocks of memory for it to run.
  • RAM has physical addresses.
  • If two processes are created (P1 & P2), the OS allocates physical memory/addresses to them.
  • Although processes have a virtual address space, there is a mapping (Paging) between physical and virtual addresses.
  • The two processes are mapped onto two different physical addresses in RAM — the OS makes sure of this.

Memory Layout of a Program

🗂️ The Segments

A program stores different types of data in different memory locations:

  • Text segment — for strings storage (code/strings).
  • Data segment — initialized global variables (e.g., x) are stored here.
  • BSS segment — uninitialized variables (e.g., y, and z if not initialized).
  • Stack — local variables (e.g., a, b, ptr).
  • Heap — the blocks of memory that a pointer (ptr) points to are stored here.

Rule of thumb: a variable goes to data if initialized, otherwise to the BSS segment. ptr lives on the stack but points to a block on the heap.

05

Stack Layout & the Stack Frame

🧱 Three Portions of a Stack Frame

When a function is invoked, three portions take place inside the stack frame:

  • Arguments (a, b) — pushed onto the stack by the caller function before jumping into the new function.
  • Special region — the bridge between the old function and the new one. It contains:
    • Return Address (4 bytes) — the location the CPU jumps to when the function finishes.
    • Old EBP (4 bytes) — the base pointer of the previous function, saved so it can be restored later.
  • Local variables — space reserved for variables declared inside the function (e.g., int x).

📍 Why Use the EBP Register

During compilation it is difficult for the compiler to get the exact location of the variables — the compiler has no idea. The problem is solved by using registers (ebp).

Since the special region has two 4-byte portions, locations become easy for the compiler to compute relative to ebp:

a = ebp + 8
b = ebp + 12 arguments addressed as positive offsets from ebp

Local variables are addressed as negative offsets, e.g. the assembly -8(%ebp) refers to the local variable x.

🎞️ Anatomy of a Stack Frame

Memory addresses grow upward here. The stack itself grows downward (high → low), but a buffer is filled upward (low → high) — keep both arrows in mind, because their opposition is the whole reason an overflow can reach the return address.

The buffer fills from the bottom upward toward the saved Old EBP and Return Address. As long as the input fits, those control values are never touched.

06

Function Invocation in the Stack

🔗 Nested Calls: main() → foo() → strcpy()

Consider main() containing foo(), which contains strcpy():

  • When main() invokes foo(), the program allocates a new block of space for foo().
  • We need ebp to point at foo()'s special region — but there is only one ebp register, needed to point to foo()'s local variables and arguments.

💾 Saving the Old EBP

  • We must remember the value of ebp to properly point to both venues.
  • The first region of foo() stores the value of ebp as (old ebp), so it can point back to main().
  • Inside foo(), strcpy() also acquires a new block of space with two arguments, and the same old ebp value is stored in strcpy()'s frame.

↩️ Return Addresses

Keep in mind the location where you must return from one function to another — save the return address of each level:

  • When strcpy() returns, we return to location (1); its space is freed and ebp jumps to the upper level.
  • When foo() returns, we return to location (2); its space is freed and ebp jumps up to main().

07

The Buffer Overflow Vulnerability

📋 strcpy() and the Null Terminator

When copying data from src to dest, we don't know how much data must be copied into dest. But one character tells us where to stop: \0 (value 0), the termination character.

If you remove \0, the compiler will copy the whole string and place \0 by itself at the end.

💥 How the Overflow Happens

  • foo() is invoked in main(), and main() passes str to it.
  • foo() has a local string (buffer) of size 12, but it copies a str of more than 12 characters into it.
  • So this buffer will overflow.
  • The whole string is copied even outside the buffer portion until its end, but the outside portion gets corrupted.
Important misconception: some think the data/buffer grows as the stack grows (high to low) — No. Only the stack grows from high to low; the buffer (data) is copied from low to high addresses, which is exactly why it overruns into the saved ebp and return address.

🎞️ Watch the Overflow Happen

The buffer holds 12 bytes, but strcpy() keeps copying until it hits \0. Watch the copy climb past the buffer's top edge and spill into the saved Old EBP and then the Return Address.

Once the saved Old EBP and Return Address are overwritten with arbitrary bytes, the function no longer knows where to return — leading directly to the crash scenarios in the next topic.

08

Why the Program Crashes

⚠️ Corrupted Return Address

When the buffer overflows, the return value/addresses are replaced with your data that spilled outside the given space. So the real address is no longer there, and the program can crash. There are several scenarios:

  • Unmapped virtual address: if you jump to a virtual address not mapped to a physical address, it's an invalid address → the OS will kill the process.
  • Invalid instructions: the address maps to an invalid place, so you may jump to invalid instructions → the CPU won't recognize them and kills the process.
  • Restricted area: if you have a valid address but it's a restricted area such as OS kernel territory, the CPU won't allow you there and the program crashes.
  • No pointer address: if you have no pointer address, the program also crashes.
09

Exploiting the Vulnerability

🎯 Why a Privileged Program is the Target

If it is a vulnerable AND privileged program — e.g. a Set-UID program running on a server with a known vulnerability — the damage can be massive. To attack the code, we try to insert our malicious code into it so we can do damage.

🧨 The Attack Idea

  • If we insert our malicious code at the top and we are unable to reach it, it's dead code.
  • Most of the stack contains data, not code — but the return address can lead us to code.
  • The idea: overflow the buffer and replace our data until it overwrites the return address.
  • We place the address of our code (A) in the return-address slot.
  • So if we can make execution jump to our code, we can run it and do damage.

🎞️ Hijacking the Return Address

Instead of random bytes, the attacker carefully overwrites the return address with the address A of the injected code. When foo() finishes, the CPU dutifully "returns" — straight into the attacker's shellcode.

Vulnerable Program & Experimental Setup

🧪 The Vulnerable Program

  • Set up the environment: a vulnerable program with a buffer of size 100.
  • main() gets its input from a file called 'badfile'.
  • main() reads 300 characters from badfile, then invokes foo().
  • Inside foo(), it copies the 300 characters (str) into the 100-character buffer.
  • Obviously there is a buffer overflow.
  • The program is a Set-UID program owned by root.

⚙️ Experimental Setup (Disabling Countermeasures)

Buffer overflow is one of the oldest vulnerabilities. The VM has several countermeasures; turn some off for this task:

  • Set memory randomization = 0 (off) — you need to be root, use sudo.
  • Compile the code and turn off two other countermeasures.
  • Make it a privileged program: change ownership & set the Set-UID bit.
  • Result: a vulnerable-privileged program.

🔬 Demonstrating the Crash

  • The stack takes input from badfile, so generate 100 bytes of random numbers into badfile — no problem occurs, since the buffer size is 100.
  • Try again with 160 bytes of random numbers into the file.
  • After executing, a fault message occurs — meaning the buffer-overflow program crashes.
11

Exploit Challenges & NOP Sled

🎯 The Goal

foo() copies data into the buffer and it overflows, writing data over the return-address section. We want to let the program return to our code — our job is to make foo() return to our code.

❓ Two Challenges

  • Where do we put the return address? → What is the offset? (the distance from the buffer start to the return-address slot)
  • What value do we put into the address? The value should be the beginning of our code. But we don't know exactly where our code is placed in the stack.

🛬 NOP Sled — Multiple Entry Points

Because we don't know the exact start of our code, we create multiple entry points to increase the chances of a correct guess, so we can enter our code from multiple points.

  • We place a lot of NOP (no-operation) instructions before our code.
  • Landing anywhere in the NOP region slides execution down into our code, so the success rate increases.
  • If we get the ebp value and the address of the buffer, we can compute the offset. We then add some value to ebp so we land on one of the NOP values.

🎞️ Why the NOP Sled Improves the Odds

A precise jump to the first shellcode byte is hard to guess. A long runway of NOP (0x90) instructions does nothing but advance to the next instruction — so any landing inside the sled "slides" the CPU forward into the shellcode. Watch three different (imperfect) guesses all succeed.

This is why the chosen return value need not be exact — ebp+8, +100, or +120 all work as long as they point somewhere inside the sled.

Finding the Offset and Addresses (GDB)

🐞 Debugging with GDB

We assume we have the source code and run it for debugging:

  • Set a breakpoint so we can stop the program during execution and print out variables.
  • When foo() is invoked, the program stops.
  • Make sure badfile exists before running GDB — create badfile first.
  • The program runs and stops inside foo(); then print the value of ebp and buffer.

📐 Computing the Offset

  • Use the p command to print the values of ebp and buffer.
  • Use p/d to get the difference of these addresses.
offset = ebp − buffer = 108 distance from buffer start to the saved ebp

This tells us the candidate value of the return address (R.T). We need to return into one of our NOPs, so ebp+8 is a candidate for R.T.

In the actual attack we may not use ebp+8; we might use +100, +120, etc., depending on the experiment's scenario.

Constructing the Payload (badfile)

🧩 Layout of the badfile

  • Construct the badfile and copy it onto the stack.
  • At the end of badfile, put our malicious code.
  • The return address 'A' must land in the NOP range.
  • Put the return address at a particular location so that, when the buffer overflows, our code is triggered. Everything is on our side — overflow the buffer and our code runs.

⚠️ The Null-Byte (0) Pitfall

Remember: strcpy() stops at 0. If strcpy() sees a 0, it stops copying — which would stop your attack.

  • Inside our payload, there should be no 0.
  • The return address may contain a 0 (placed at the end so copying still reaches it).

Exploit Code & Launching the Attack

💻 Shellcode & Exploit

  • Machine code is put inside the shellcode.
  • Once the code is ready, put it into badfile and launch the attack.
  • Prepare exploit.py and the stack program.
  • Run exploit.py → generates badfile.
  • Run stack → you get a shell.

⚠️ First Attempt Fails to Get Root

Running stack gives a shell, but not a root shell. Checking id shows we don't have root privilege — something went wrong.

15

The Countermeasure Issue (sh → zsh)

🛡️ A Linux Countermeasure

The issue is one of the countermeasures in the Linux OS, in the shell program. We change it so the shell program does not have the countermeasure:

  • Link sh with zsh.
  • Now do the attack again.

👑 Getting Root

Run stack → the prompt sign changes from $ to # (indication of root). Check id → euid=0.

After the experiment, revert the countermeasure, because many other things depend on it.

📝 Common Exam Questions
  • Define a buffer overflow. → Putting more data into a fixed-length buffer than it can hold, so the excess overwrites adjacent memory
  • Name the program memory segments and what each stores. → Text (code/strings), Data (init globals), BSS (uninit globals), Heap (dynamic blocks), Stack (locals)
  • What are the three portions of a stack frame? → Arguments, special region (return address + old ebp), and local variables
  • How are arguments and locals addressed relative to ebp? → Arguments at positive offsets (a=ebp+8, b=ebp+12); locals at negative offsets (e.g., -8(%ebp))
  • Why is the old ebp saved? → There is only one ebp register; saving it lets the previous function's frame be restored on return
  • Where does strcpy stop and why does that matter for an exploit? → It stops at a 0 byte, so the payload must contain no 0 bytes (the return address may)
  • Why does an overflowed program crash? → The return address is overwritten; jumping to an unmapped/invalid/restricted address makes the OS or CPU kill the process
  • What is the attack idea? → Overflow the buffer to overwrite the return address with the address of attacker-supplied code
  • What is a NOP sled and why use it? → A run of NOP instructions giving many entry points, so an approximate return address still slides into the shellcode
  • How is the offset found in the lab? → In GDB, print ebp and buffer and take their difference (p/d), e.g., 108
  • Why did the first exploit not yield root, and how was it fixed? → A shell countermeasure dropped privileges; linking sh to zsh bypassed it, giving euid=0 (prompt $ → #)
  • Which countermeasures are disabled in the setup? → Memory (address) randomization plus two others; then ownership is changed and the Set-UID bit is set
